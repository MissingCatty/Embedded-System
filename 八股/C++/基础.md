# 1.结构体内存对齐

结构体成员按照声明的顺序进行存储，但默认的对其方式是按照成员中最大的那个大小对其。

比如，某个结构体中最大的成员为double类型，那就是按照8字节的方式对齐。

## 1.1 c++对齐控制

### 1.1.1 alignas

**功能**：返回某个类型的**对齐字节数（alignment requirement）**

```cpp
#include <iostream>

int main() {
    std::cout << alignof(int) << std::endl;
    std::cout << alignof(double) << std::endl;
    std::cout << alignof(char) << std::endl;
}
```

输出：

```cpp
4
8
1
```

### 1.1.2 alignof

**功能**：显式指定变量或类型的对齐方式。

```cpp
alignas(16) int x;	// 显式指定变量x按照16字节对齐
```

## 1.2 案例

```cpp
struct A {
    char a;
    alignas(16) int b;
};
```

- 变量a：1字节
- 变量b：4字节，且16字节对齐（结构体内offset为16的倍数）

```
| a(1B) | padding(15B) | b(4B) | padding(12B) |
    0         1~15       16~19	       20~31
```

- 结构体总大小为：32字节

# 2. C++代码从编辑到运行需要进行哪些步骤

## 2.1 预编译（.c 到 .i或.ii)

处理#include，#define等以#开头的预编译代码，生成一个准备好进行编译的中间文件。

- 通过文本替换对#define进行展开
- 处理所有条件编译（#if，#ifdef，#endif，#elif，#else）
- 将被#include修饰的包含的文件递归展开
- 删除注释
- 保留#pragma编译器指令，编译器需要这些指令控制编译过程。（如#pragma once防止编译时被重复包含）
- 添加调试信息（在源文件中添加行号和文件标识）

## 2.2 编译

编译阶段将预编译生成的中间文件编译成汇编代码（.s）。

## 2.3 汇编

将汇编代码翻译成机器码，生成目标文件（.o或.obj）

## 2.4 链接

存在多个源文件的时候，需要将多个源文件对应的目标文件，链接成一个完整的可执行文件。

> 其实编译阶段就已经翻译成机器码了，只不过有些符号的地址没有确定（多个文件存在并共享同一个符号，即全局变量和函数），链接的过程就是确定符号地址的过程。

- 符号解析

  - **术语**

    - **符号**：源文件中所有**函数名**、**全局变量名**
		> 对于编译器和链接器而言，它们不认识你代码里的逻辑，它们只认标识符。
		> 当编译器处理源代码时，会把这些**全局变量名**和**函数名**提取出来，存放在一个叫做**符号表（Symbol Table）**的数据结构中。

    - **定义**：这个符号在物理（或虚拟）内存中**真正占据的空间和具体内容**。
		> **变量的定义：** 就是在内存的特定区域（如数据段或 BSS 段）实实在在地划分出了一块特定大小（比如 4 字节）的内存空间。
		> **函数的定义：** 就是这个函数内部的代码逻辑被翻译成**机器指令（Machine Code）**后，在内存（代码段）中占据的那一段连续的字节指令。

  - **过程**：

    - 在编写大型程序时，我们经常在一个文件（比如 `A.cpp`）里使用了一个**符号**，但它的**定义**（实际的内存空间或机器指令）其实在另一个文件（比如 `B.cpp`）里。
    - 编译 `A.cpp` 时，编译器只看到了符号，但找不到定义，所以它会在 `A.o` 的目标文件中留下一个“未解析的引用（Unresolved Reference）”。
    - **链接器的任务，就是拿着 `A.o` 里的这个“符号名字”，去 `B.o`（或者其他的库文件）里寻找与之匹配的“实体定义”，然后把它们缝合在一起。**

- 地址重定位

  编译器生成目标文件时，代码里的内存地址通常是**从 0 开始的相对地址**。

  链接器在组合这些文件时，会为程序中的每个函数和变量分配最终**在实际内存中运行时的绝对（或虚拟）地址**。

  随后，它会修改目标文件里所有引用这些符号的占位符，将其**替换为真正的绝对地址**，确保程序内部的相互调用正确无误。

程序在链接库文件时，根据“合并代码的时机”不同，分为静态链接和动态链接。

- 静态链接：静态链接库（**linux中的.a和windows中的.lib**）本质上就是一个压缩文件，其中包含了打包后的（.o）文件。如果用户代码在程序中引用了一个静态库中的函数，那链接的时候就要到库文件中找到函数的定义，此时就要到静态库文件（.a）中查找符号表，找到包含该函数定义的.o文件，和用户代码对应的.o文件一起执行链接。

- 动态链接：

  - **编译阶段**：当你在编译完成后执行链接操作时，链接器（Linker）依然会介入，但它对待动态库（如 `.so` 或 `.dll`）的态度非常“敷衍”：

    - **绝对不拷贝机器码：** 链接器不会把动态库里哪怕一字节的指令合并到你的可执行文件里。
    - **记录依赖清单：** 它只会在你的可执行文件头部写下一条记录：“注意，这个程序在未来运行时，必须带上 `libA.so` 和 `libB.so` 这几个库”。
    - **留下查找表和存根：** 对于你调用的外部函数（比如 `printf`），链接器会在你的程序里生成一个小型的查找表（GOT，全局偏移表）和一段跳转存根（PLT，过程链接表）。此时，查找表里的地址是空的或者无效的，因为它现在根本不知道动态库将来会被加载到内存的哪个角落。

    经过这一步，生成的可执行文件非常小，因为它只包含你自己写的机器码和一堆“指路牌”。

  - **运行阶段：**真正的地址合并，是在你双击运行程序的那一瞬间（或者运行中）才发生的。

    - **操作系统加载：** 当你运行程序时，操作系统首先把你的可执行文件（机器码）加载进内存。
    - **动态链接器登场：** 操作系统读取了刚才留下的“依赖清单”，发现你需要外部库，于是它会唤醒系统里的**动态链接器（Dynamic Linker/Loader，例如 Linux 下的 `ld-linux.so`）**。
    - **加载共享库：** 动态链接器去系统的硬盘路径下找到那些 `.so` 文件，把它们也读取到物理内存中。如果内存里之前已经有其他程序加载过这个库了，就直接共享那一块内存，不再重复读取。
    - **运行时重定位（填坑）：** 动态链接器算出了刚刚加载的动态库在内存里的真实基地址。然后，它去修改你程序里留下的那个查找表（GOT），**把真实的内存绝对地址填进去**。

# 3.可执行程序的后缀，在内存中的开始地址

Linux 不通过扩展名（如 `.exe`）来判断文件是否可执行，而是依赖文件头部的魔数（Magic Number，比如 ELF 文件的 `\x7fELF`）以及文件的执行权限（`chmod +x`）。

ELF（Executable and Linkable Format）确实是 Linux 系统下标准的二进制目标文件格式，无论是可执行文件、动态链接库（`.so`）还是目标文件（`.o`）都采用这种格式。

加载可执行文件的时候，操作系统会将其加载到内存的特定位置。

这个位置的起始地址取决于操作系统的内存布局和系统架构。

出于安全考虑，现代编译器（如 GCC、Clang）默认会开启 `-fPIE` 和 `-pie` 选项，将所有的可执行文件编译为**位置无关代码**。这意味着程序内部不再硬编码像 `0x400000` 这样的绝对地址，而是全部使用相对地址（RIP-relative addressing）。

# 4.内存空间划分

可执行程序的内存空间包含一下几种：

- 代码段（.text）：用于存放可执行代码
- 数据段（.data）：存放**已初始化的全局变量**和**静态变量**
- bss段（.bss）：存放**未初始化的全局变量**和**静态变量**
- 堆：动态分配内存的区域（malloc或new）
- 栈：局部变量和函数调用
# 5.解释一下`int main(int argc, char *argv[])`

- `int argc`：表示在执行main函数的时候，命令行传入了多少个参数，包含可执行程序名。
	- 如：`./program arg1 arg2`的`argc`为3
- `char *argv[]`：保存输入指令的字符串，其长度为`argc+1`，因为这个数组的最后一个元素是`NULL`，指示数组的结尾。
# 6.函数调用在内存中的过程

首先，我们需要知道：一个线程中的所有函数，都共享一个巨大且**连续的栈**。

每次发生函数调用的时候，其都会在这个共享的大栈上划分出属于自己**栈帧**。

![](C:/Users/13566/Desktop/assets/CPP/file-20260215194148867.jpg)

现在问题来了：**如何确定每个函数自己的栈的范围？**

当前函数栈的范围由两个寄存器EBP（栈基址指针）和ESP（栈顶指针）划出。

---
**从发生函数调用到返回当前函数期间，栈中数据如何变化？**

```c
int substract(int a, int b, int c)
{
	int res = a-b-c;
	return res;
}

int main()
{
	substract(10,20,30);
	return 0;
}
```

- 阶段一：保存参数和返回地址
	- 【参数】入栈：在main中调用`substract`函数，首先会在栈中从右到左依次入栈`30,20,10`
	- 【main函数返回后下一条指令地址】入栈
- 阶段二：切换栈底并为局部变量分配空间
	- 【main函数EBP】入栈：旨在保存main函数栈的基址
	- 切换EBP为`substract`函数的基址
	- 【res变量分配空间】：旨在为局部变量分配空间
- 阶段三：打扫战场并返回
	- 销毁局部变量：出栈
	- 恢复main函数栈底
	- 跳回主函数
	- 清理参数

由此可见：各个数据入栈和出栈的顺序的一一对应的。

# 7.const关键字

常量的本质就是一种编译器层面的约束，仅仅是在编译阶段进行检查，如果检查不通过，则拒绝编译生成机器码。

这是在栈上保存的**局部变量**，使用const在编译层面上进行约束；

> 还有一种是在**硬件层面上**的修改保护：
>
> 在 ELF 可执行文件（Linux）或 PE 文件（Windows）中，`.rodata`（Read-Only Data，只读数据段）是操作系统重点保护的“禁区”。当程序加载到内存时，这块区域会被 MMU（内存管理单元）强制打上**只读（Read-Only）的权限标签**。
>
> 会被放到这个内存段的数据包含：
>
> - **字符串字面量**
>
>   例如，`str = "Hello World"` 就静静地躺在 `.rodata` 段里。
>
>   如果执行`str[0] = 'h'`，则会触发操作系统硬件的写保护机制，禁止对该内存进行修改。
>
> - **全局与静态常量**
>
>   即保存在全局区的常量（局部常量属于纸老虎，只由编译器保护）
>
>   ```c
>   /* 全局常量 */
>   const int x = 10;
>   int main(void){
>       /* 静态常量 */
>   	static const int y = 12;
>   }
>   ```

## 7.1 修饰变量

用于定义常量，被修饰的变量初始化后不可修改。

最容易混淆的是常量指针和指针常量：

- 常量指针：顾名思义，就是指向一个常量的指针，不能修改其指向的内容。
```c
const int *p = &a;
*p = 10;  // 错误
```

- 指针常量：顾名思义，就是该指针是一个常量，即指向的地址一旦被确定就不能修改。
```c
int* const p = &a;
```

记法：
- `const int* p`中的`*`修饰的是`const int`，代表该指针指向的是一个常量。
- `int* const p`中的`*`修饰的是int，本质是一个指向int类型的指针。

## 7.2 修饰类成员函数

在类的成员函数后面加上const，表示该函数不会修改任何成员变量。

```c
class MyClass{
	int data:
public:
	int getData() const {
		return data;
	}
}
```

如果一个对象被const修饰，说明其中的成员变量不可变，即可以调用被const修饰的成员函数。

```c
const MyClass myClass;
myClass.getData();
```

## 7.3 如何修改常量的值

要接触编译器层面的约束，则也要使用编译器层面的手段进行解除。

### 7.3.1 const_cast

【注意】：不推荐使用，对于存储在`rodata`里的常量（全局或静态常量），如果让编译器放行写保护，则可能会导致修改后内存管理报错。

`cast_const`：C++操作符，用于移除对象的const限制。

```c
const_cast<新类型>(表达式);
const int x = 10;
int *p = const_cast<int*>(&x);
*p = 12;
```

- 新类型必须是指针类型或引用

### 7.3.2 memcpy

直接对内存进行修改，风险较高，不推荐使用。

```c
const int x = 10;
int* y = &x;	// 这一步在编译时就报错，因为编译器会认为将const int*赋值int*会造成不安全
*y = 11;
```

```c
const int x = 10;
int y = 20;
memcpy((void*)&x, &y, sizeof(int));	// 加上void*的目的是强行撕掉const int*的写保护
```

## 7.4 const变量值何时确定

- 编译时确定

  ```c
  const int* x = 10;
  ```

- 运行时确定

  ```c
  const int* x = getNum();
  ```

区别：编译期可以对编译时确定的变量优化，当运行时的不行。
