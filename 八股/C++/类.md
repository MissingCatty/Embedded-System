# 1.指针和引用的区别

| 特性                 | 指针                                   | 引用                                   |
| -------------------- | -------------------------------------- | -------------------------------------- |
| 本质                 | 存储一个地址                           | 是原变量的别名                         |
| **级别**             | 可以有多级指针                         | 只有一级引用                           |
| **是否可为空**       | 可以为空                               | 不能为空，必须初始化                   |
| **初始化后是否可变** | 可以改变指向                           | 不可改变引用的目标                     |
| **sizeof 操作**      | 返回指针的大小（地址的大小）           | 返回引用所指向的变量的大小             |
| 作为函数参数传递     | 传递的是指针的拷贝，改变指向不影响实参 | 传递的是对原变量的引用，改变会影响实参 |
| **内存占用**         | 占用存储地址的大小（通常为 4/8 字节）  | 占用与指针相同的内存                   |
| 声明和初始化         | 可以先声明再初始化                     | 必须在声明时初始化                     |
| 是否可以指向空值     | 可以指向 nullptr 或空值                | 不能指向空值                           |

**引用的本质**是：编译器在符号表中对同一段内存赋予一个新的符号。

# 2.什么情况使用引用和指针

引用：

1. 函数传参传递的是类对象时：在函数中对对象的修改直接作用于原对象，而不需要创建副本。

指针：

1. 传入的对象可能是null的时候，因为引用不能为空

# 4.变量声明和定义的区别

声明和定义的核心在于是否分配了内存。

- **声明**：告诉编译器这里有这个变量，但不分配内存，多用于多个文件之间通过`extern`共享变量，使链接器能够正确的找到变量。
- **定义**：创建变量并分配内存。

# 5.strlen和sizeof的区别

- strlen是库函数，在运行时计算内存，但只适用于**字符串**和**以`/0`结尾的字符数组**。
- sizeof是**运算符**，只能用于计算静态分配的内存大小，因为sizeof是在编译的时候就算出字面量。

# 6.C++中调用C语言

## 6.1 问题复现

不能再C++中直接调用C语言的相关函数，主要矛盾出在C++和C语言的符号不匹配上。

- `math.c`：该文件由C语言编写

  ```c
  // 这是一个纯 C 语言文件，用 gcc 编译
  int add(int a, int b) {
      return a + b;
  }
  ```

- `main.cpp`：由C++编写

  ```c
  #include <iostream>
  
  int add(int a, int b); 
  
  int main() {
      int result = add(10, 20);
      std::cout << result << std::endl;
      return 0;
  }
  ```

- 分别编译两个文件：

  ```bash
  gcc -c math.c -o math.o   # 正常通过
  g++ -c main.cpp -o main.o # 正常通过
  ```

- 执行链接

  ```bash
  g++ main.o math.o -o program
  ```

  当场报错

  ```bash
  main.o: In function 'main':
  main.cpp:(.text+0x14): undefined reference to 'add(int, int)'
  collect2: error: ld returned 1 exit status
  ```

明明在`math.c`中定义了`add`函数，为什么编译器报错说找不到呢？

因为在执行编译命令的时候，会分别为`math.c`和`main.cpp`生成两个文件的符号表，符号表里包含了函数对应的符号名和其地址。

- 在`math.c`对应的符号表中（C语言），`add`函数的符号就是原封不动的`add`；
- 而在`main.cpp`中的符号表中（C++），`add`函数的符号除了由函数名组成，还由参数列表组成（因为C++支持重载），其符号名可能是`_Z3addii`

所以在链接阶段，由于C++和C语言的符号生成规则不同，导致符号匹配不上，所以就找不到定义。

## 6.2 问题解决

找到了问题所在，我们只需要让编译器知道声明的`add`函数的符号查找规则还是按照老一套的C语言的查找规则进行即可。

```c
#include <iostream>

// 加入抢救神药：extern "C"
extern "C" {
    int add(int a, int b); 
}

int main() {
    int result = add(10, 20);
    std::cout << result << std::endl;
    return 0;
}
```

## 6.3 企业级的使用示例

C++编译器自带内置宏`__cplusplus`，如果使用的是C++编译器，该宏就会被开启。利用这点，我们就可以很好的对代码进行适配：

```cpp
// 文件：c_math.h

#ifdef __cplusplus
extern "C" {      // 如果是 C++ 编译器在看这段代码，就加上 extern "C" {
#endif

// 这里写所有的 C 函数声明
int add(int a, int b);
void do_something();

#ifdef __cplusplus
}                 // 如果是 C++ 编译器，别忘了闭合大括号 }
#endif
```

即：对于所有可能涉及到用C语言编写的函数，声明的时候要加上`extern "C" {...}`。

# 9.内联函数

当你在函数前面加上 `inline` 关键字：

```c
inline int add(int a, int b) {
    return a + b;
}
```

你其实是在向编译器提议：**“大哥，这个函数太短了，别搞压栈跳转那一套虚头巴脑的了。不管我在哪调用它，你直接把这几行代码给我‘复制粘贴’到调用的地方去！”**

于是，当你在主函数写下 `int c = add(10, 20);` 时：

- 编译器在底层生成机器码时，根本不会生成 `call add` 这样的跳转指令。
- 它直接把你那句代码“强行展开”，在底层等价于变成了类似 `int c = 10 + 20;` 的直接运算。

【注意点】：

- 空间换时间
- 不要以为加了 `inline`，编译器就一定会乖乖听话。现代编译器非常聪明，它有自己的脾气：
  - 如果你的函数**非常短小**（比如一两行简单的赋值或运算），编译器会欣然接受，帮你展开。
  - 如果你的函数**很长**，或者里面有复杂的 `while/for` 循环，甚至包含**递归**调用。编译器心里就会骂娘：“这么一大坨你让我到处复制粘贴？” 然后它会**直接无视你的 `inline` 关键字**，强行把它当成一个普通函数去压栈调用。
- 内联函数必须写在头文件里（函数体也要写在头文件里）

# 12.volatile、mutable和explicit关键字

## 12.1 volatile

告诉编译器，被`volatile`关键字修饰的变量必须从**内存**读取，而不是从**寄存器**读取。

不加`volatile`时，程序**单线程完美，多线程灾难**。

【情景一】：不加`volatile`

```c
int flag = 0; // 普通变量

void wait_for_flag() {
    while (flag == 0) {
        // 傻等，什么都不干
    }
}
```

编译器开启 -O2 优化，编译得到的汇编代码如下：

```assembly
wait_for_flag:
	; DWORD PTR flag[rip]是物理内存
    mov eax, DWORD PTR flag[rip]  ; 【致命一步】只在进循环前，去内存读一次 flag，存进 eax 寄存器
    test eax, eax                 ; 检查 eax 是不是 0
    jne .L1                       ; 如果不是 0，直接结束函数
.L2:
    jmp .L2                       ; 🔴 【灾难发生】如果是 0，直接进入死循环！再也不会去内存读数据了！
.L1:
    ret
```

也就是说编译器先从内存中读取目标变量到高速寄存器里（好比先把字典中的数据抄到小本本上），之后对该变量的所有读取和修改操作全是在这个告诉寄存器里进行。

- 好处：对于单线程的程序来说，这是完美的，因为只有该线程会访问和修改这个变量。
- 坏处：对于多线程来说，某一刻其他线程修改了变量之后，也放到自己的寄存器，那这个线程拿到的值就不匹配了。

所以，对于共享变量使用`volatile`关键字修饰，就是告诉编译器，这两个变量必须从内存读取，不要放到cpu内部的寄存器里读取。

如果一个类的**非静态成员变量**被mutable修饰，即使该类的对象被`const`修饰，那么该变量也是可以修改的。

## 12.3 explicit

首先咱们来看一下字符串的赋值：

```cpp
std:string str = "abcd";
```

这里的`"abcd"`不是`string`类型，他们在底层只是一个纯C的`const char*`。之所以能够直接给C++的字符串类型赋值，并且生成一个`string`类对象，是因为C++编译器规定：**一个类如果有一个只包含一个参数的构造函数，就能通过直接赋值的方式，把数据直接赋值给这个参数进行构造，最后隐式创建一个对象。**

但这种特性在提供方便的同时，也造成了麻烦：

```cpp
#include <iostream>

class MyArray {
public:
    int m_size;
    // 没加 explicit 的单参数构造函数
    MyArray(int size) : m_size(size) {
        std::cout << "开辟了大小为 " << size << " 的数组\n";
    }
};

// 一个处理数组的普通函数
void processArray(MyArray arr) {
    std::cout << "正在处理数组...\n";
}

int main() {
    MyArray arr = 10;  // ❌ 荒谬：一个数组怎么能“等于”数字 10？
    
    processArray(20);  // ❌ 更荒谬：函数明明要求传入一个 MyArray 对象，你却传了个数字 20！
    
    return 0;
}
```

在构造函数上加上`explicit`关键字，就会**禁止进行隐式构造**。

# 14.static的用法和作用

1. 限制变量和函数的作用域
