# 3.new和malloc的实现原理

- malloc

  malloc底层是一个内存池管理器。当程序使用`malloc`分配内存时，glibc有一个默认的阈值`M_MMAP_THRESHOLD = 128kb`。

  - 当需要分配的内存小于`128kb`时，malloc会调用底层的`brk()`函数（把进程虚拟内存空间中的“堆区”的最高地址堆顶指针往上推一点）
  - 当分配的内存大于`128kb`时，`malloc` 觉得这玩意儿太大，放在堆里容易搞乱现有的空间，于是它会调用 **`mmap()`**（内存映射）。`mmap` 会绕过传统的堆区，直接在广阔的“内存映射区”里，单独给你开辟一块巨大且独立的匿名物理内存。这块内存用完后，调用 `munmap()` 就能立刻还给操作系统，干脆利落。

  同时，malloc使用`brk`分配得到的内存，会将其分成若干小块（一个个带有header的chunks），这些chunks用一个隐式链表组织起来。

  ```c
  void* p = malloc(24);
  ```

  `malloc` 绝对不会只给你 24 字节。它会在你要求的内存前面，强制加上一个 **头部 (Header)**（通常是 4 字节或 8 字节）。

  这块地皮的真实物理切片是这样的： `[ 头部 Header (8字节) ] + [ 你的数据 Payload (24字节) ]`。

  这个Header把一个 64 位的整数劈成了两半用：

  1. **高 63 位：记录这整块内存的大小 (Size)。**（在这里就是 8 + 24 = 32 字节）。
  2. **最低的 1 位：记录这块内存的分配状态 (Allocated Bit)。** `1` 代表已经被分配（占用），`0` 代表这块地是空闲的。

  > 假设整个堆区像一条长长的街道，挨个挤满了房子（内存块）。`malloc` 站在街头（堆区的起始地址），手里拿着你要申请的 24 字节订单。
  >
  > 1. **第一跳：** 它读取第一栋房子的 Header。发现最低位是 `1`（被人占了），大小是 `64`。
  > 2. **施展黑魔法（寻找下一栋）：** 它根本不需要找 `next` 指针，它直接把**当前的物理地址加上 `64`**！这就像一个完美的跳远，精准地落在了第二栋房子的 Header 上。
  > 3. **第二跳：** 读取第二栋房子的 Header。发现最低位是 `0`（空闲！），大小是 `16`。可惜，虽然空着，但 16 字节不够你塞 24 字节的数据。
  > 4. **继续跳：** 当前地址再加 `16`，精准跳到第三栋房子的 Header……

  > **为什么要把malloc得到的内存切成一个个chunk？**
  >
  > 1. 方便回收
  >
  >    申请内存用的是 `void* p = malloc(24);`，你明确告诉了系统你要 24 个字节。 但是，当你释放内存时，你写的是 **`free(p);`**
  >
  >    发现盲点了吗？`free` 函数只有一个指针参数，它根本不知道你要释放多少字节！
  >
  >    因为隐式链表的每块内存的header里都存放了这块内存有多少个字节。
  >
  > 2. 避免内存碎片
  >
  >    假设内存不分 Chunk，只是一条连续的白纸。 程序运行了一会儿：
  >
  >    - 分配了 10 字节。
  >    - 分配了 50 字节。
  >    - 分配了 20 字节。 然后，中间那个 50 字节的被释放了。此时内存里出现了一个 50 字节的“窟窿”。
  >
  >    如果有人接着申请 60 字节，这个窟窿塞不下；如果有人申请 10 字节，塞进去之后又剩下一个 40 字节的更小窟窿。久而久之，你的物理内存会被撕得粉碎，到处都是几字节、十几字节的缝隙，虽然总剩余空间很大，但**连一块连续的大内存都掏不出来**。
  >
  >    **Chunk 的破局（合并机制 Coalescing）：** 因为有了 Chunk 和 Header，系统可以精准地掌握每一个“窟窿”的大小和位置。 当我们释放一个 Chunk 时，系统会顺着相邻的 Chunk Header 检查：“哎，我左边的 Chunk 也是空闲的，右边的也是空闲的！” 于是，系统会**瞬间把这三个小 Chunk 的 Header 抹掉，物理合并成一个巨大的空闲 Chunk**。这完美地抵抗了碎片的产生。

- new

  new的本质是`operator new + 构造函数`，底层调`malloc`。

- delete

  delete的本质是`析构函数 + operator delete`，底层调`free`。

# 4.被free掉的内存是立刻返还给操作系统吗

C++ 程序员最容易产生错觉的一个问题：**我明明调用了 `free`，为什么在 Linux 的 `top` 命令里看，我进程的内存占用根本没有降下来？**

**原因：`free` 绝大多数情况下根本不会把内存还给操作系统！**

- **真正的归属：** 当调用 `free` 回收内存时，内存并不会立即返还给操作系统，而是被保留在用户进程的堆区中，由内存分配器（如 `ptmalloc`）管理。
- **为什么这么做（核心动机）：** 这样做可以避免频繁的系统调用，从而提高内存分配效率并减少系统开销。每次向操作系统内核伸手要钱（系统调用）都会发生极其昂贵的用户态到内核态的上下文切换，`ptmalloc` 作为精明的二房东，当然要把退租的房子自己捏在手里，等下一个租客来。

**free后的动作：**

- **内部消化（再次分配）：** 当进程需要分配新内存时，分配器会优先尝试从空闲链表中找到一块合适的内存直接使用，无需向操作系统申请。
- **抵抗碎片（碎片合并）：** 为了避免内存碎片化，分配器会尝试合并相邻的空闲内存块，形成更大的连续区域。

# 5.delete p、delete p[]、allocator

- `delete p`：释放通过`new`运算符分配的单个对象的内存

- `delete p[]`：释放通过`new []`分配的动态数组

  ```cpp
  int* arr = new int[5];
  ```

- `allocator`：标准库提供的一个内存分配的工具类，将内存分配和对象构造分离。

  `new`将内存分配和对象创建两个耦合到一起，且C++中一般不使用`malloc`，而`allocator`的目的就是替代`malloc`的。

  > 举个例子
  >
  > ```cpp
  > std::vector<MyClass> vec;
  > vec.reserve(100); // 我只是想提前预留 100 个坑位，现在还不放数据
  > ```
  >
  > 调用`reserve`的时候，只是提前预留，等于说只是预留内存，对象的创建并不在此时进行。因为如果在预留阶段`new`创建对象的话，要调用100次构造函数，会极大拖缓执行速度。
  >
  > 因此，vector底层使用的是`std::allocator<MyClass>`直接分配内存。
  
  下面我们来使用allocator来模拟new创建对象的过程：
  
  ```cpp
  #include<iostream>
  #include<memory>
  #include"student.h"
  
  using namespace std;
  
  int main(void)
  {
  	allocator<Student> alloc;
  	
  	// 分配5个Student大小的内存
  	Student *p = alloc.allocate(5);
  	
  	// 在内存上构造对象
  	alloc.construct(p, 1, "jason", 12);	// 创建第一个对象
  	alloc.construct(p+1, 2, "peter", 32);	// 创建第二个对象
  		
  	// 使用对象
  	p->print();
  	(p+1)->print();
  	
  	// 销毁对象
  	alloc.destroy(p+1);
  	alloc.destroy(p);
  	
  	// 释放内存
  	alloc.deallocate(p, 5);
  }
  ```


# 6.new/delete和malloc/free的异同

| 特性         | new/delete                              | malloc/free                    |
| ------------ | --------------------------------------- | ------------------------------ |
| 语法         | **运算符**                              | **标准库函数**                 |
| 内存大小计算 | **自动计算**                            | **手动计算**                   |
| 类型安全     | **类型安全，返回指定类型指针**          | **非类型安全，返回 void*指针** |
| 构造与析构   | **调用构造函数与析构函数**              | **不调用构造函数与析构函数**   |
| 封装关系     | **封装 malloc，通过 operator new 调用** | **无封装**                     |
| 错误处理     | 抛出异常 std::bad_alloc                 | 返回 NULL                      |
| 库依赖       | 无库依赖                                | 需要标准库                     |

# 7.内存泄露、内存越界

- 内存泄漏：申请了一块内存，但使用完成后没有正确释放，导致系统或内存管理器没办法回收和重复利用内存。

  - 原因：

    1. 使用new或malloc之后没有调用delete或free进行释放

    2. 父类指针指向子类对象，且父类的析构函数没有使用`virtual`修饰，会导致在调用析构函数的时候使用的是静态绑定，会调用指针类型的析构函数，导致子类的析构函数没有被调用。

       ```cpp
       class Person {
       public:
       	Person();
       	Person(std::string gender, int age_);
       	~Person();
       private:
       	std::string gender_;
       	int age_;
       };
       
       Person *person = new Student();
       delete(person);
       ```

       输出：

       ```bash
       [Person]的默认构造函数被调用.
       [Student]的默认构造函数被调用.
       [Person]的析构函数被调用.
       ```

       可以看出父类`Person`的析构函数没有加`virtual`，导致析构函数是静态绑定，会直接调用引用类型的析构函数，也是就`Person`类的析构函数。
    
  - 如何检查

    1. 养成良好的习惯：new和delete，malloc和free必须配套使用
    2. **自定义内存管理**：将申请的内存指针用数据结构进行管理（比如链表），程序结束时检查未释放的内存块。
    3. **使用智能指针**：`std::shared_ptr`和`std::unique_ptr`会帮助自动管理内存的分配与释放。

- 内存越界：程序访问了他未被分配到的内存，会导致未定义的行为，引发程序崩溃或错误结果。

  - 常见原因：
    1. 数组访问越界
    2. 指针操作错误
       - 指针未初始化（野指针）
       - 悬空指针（指针指向的地址空间被释放，但指针未被置`NULL`）
    3.    错误的内存分配
       - 内存未分配就使用
    4. 栈溢出
  - 如何避免
    1. 使用时检查边界条件，防止指针

# 8.Flash和RAM空间

- **Flash 存储器**是一种非易失性存储器，断电后数据不会丢失。程序的代码通常存储在 Flash 中。在系统启动时，代码从 Flash 中加载到 RAM 中执行。在嵌入式系统中，某些需要长期保存的数据也可以存储在 Flash 中，例如配置文件和校准数据。
- **RAM 存储器**是一种易失性存储器，断电后数据会丢失。栈通常存储在 RAM 中，因为栈需要快速的读写操作，RAM 的高访问速度可以满足需求。运行数据存储：包括全局变量、静态变量以及动态分配的内存空间（如堆）。缓存：用于存储临时数据，支持程序运行中的高效计算和处理。

# 9.堆和栈的区别

| 特性         | 堆                                                           | 栈                                                           |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 管理方式     | 堆中的资源需要程序员手动管理，使用 new 和 delete 进行分配和释放。如果释放不当，容易造成内存泄漏（Memory Leak）。 | 栈中的资源由编译器自动管理，分配和释放都由编译器控制，不需要手动管理，使用方便且不易出现资源泄漏问题。 |
| 内存管理机制 | **系统维护一个记录空闲内存地址的链表**。当程序申请内存时，系统遍历链表找到足够大的空闲内存块，分配给程序。如果内存块大于所需，系统将多余部分重新记录到链表中。 | 栈使用的是**固定大小的连续内存空间**。如果剩余栈空间足够大，则直接分配，否则报出栈溢出错误。栈的管理机制类似于数据结构中的“先进后出”，在函数调用时，分配和释放操作有序进行，效率高。 |
| 空间大小     | **堆的空间不连续**，大小由系统中有效虚拟内存决定（32 位系统理论上最多 4GB）。因此，堆的空间灵活且较大，适合动态分配大对象。 | 栈的空间是连续的，大小通常由操作系统预设（例如 Windows 下默认栈大小是 1MB 或 2MB，可在编译时调整）。由于栈是连续的，其大小受限，不适合分配过大的数据结构。 |
| 碎片问题     | 频繁的 new 和 delete 操作会在堆中产生碎片，降低程序的内存分配效率。 | **栈是连续分配的，不会产生碎片。**其内存分配和释放机制类似于“数据结构中的栈”，先进后出，内存分配和释放一一对应，避免了碎片化问题。 |
| 生长方向     | **堆向高地址方向增长**                                       | **栈向低地址方向增长**                                       |
| 分配方式     | 堆中的内存分配完全是动态的，没有静态分配的堆。通过 new 进行分配，通过 delete 释放内存。 | 静态分配：如局部变量由编译器完成分配和释放。<br />动态分配：如 alloca 函数进行分配，资源由编译器自动释放，无需程序员干预。 |
| 分配效率     | 堆内存分配由 C/C++ 的函数库实现，底层机制复杂，**分配和释放效率较低**。 | 栈是系统提供的结构，CPU 对栈有专门的底层支持，如专用寄存器存储栈指针，分配和释放操作简单高效，效率远高于堆。 |
| 适用场景     | 适合用于需要动态分配大块内存或者生命周期与函数调用无关的对象（如动态数据结构、缓存等）。 | **适合用于函数调用栈或局部变量分配。**分配和释放由编译器控制，适合临时数据和生命周期较短的数据。 |

> - `alloca`函数用于在运行时在栈上申请内存，作用域结束自动释放
>
>   ```cpp
>   void* p = alloca(100);
>   ```
>
>   但该函数很少用，因为不安全，可能导致栈溢出。

# 10.堆快还是栈快

栈比堆快。

1. 栈的操作由**操作系统提供底层支持**，**为栈提供专门的寄存器来存放栈指针**，栈的入栈和出栈操作非常简单，**主要是指针的加减**。同时这些操作可以用专用指令来完成，效率非常高。

2. **栈空间的分配与释放是自动进行的**，无需程序员自行释放。且**分配是连续的**，不存在内存碎片的问题。而堆的分配是用C++库函数实现的，**堆内存的分配需要通过算法寻找一块足够大的空闲内存块**，消耗额外的时间。

   > 栈空间的分配和释放比较简单，就是指针的加减，并且分配的内存都是连续的。
   >
   > 堆空间需要通过算法查找符合要求的空闲块。

3. 栈中数据存储在连续的内存区域，直接根据栈指针寻址，效率高（**一次寻址**）。而要找到堆中的数据，首先要找堆内存的地址，再通过该地址找到实际分配的内存空间（**两次寻址**）。

4. 栈机制简单，而堆需要复杂的内存管理。如：记录空闲内存的链表或内存池，并在分配后进行碎片的合并。

# 11.C++中类的数据成员和成员函数的内存分布

**数据成员的内存分布**

C++类是从C的结构体发展过来的，因此其成员变量的内存分配机制与C的结构体类似。

- 成员变量：类中的每个成员变量，其存储顺序都由声明顺序决定，并且在内存分配过程中受内存对齐的影响。
- 类对象首地址：首地址为第一个成员变量的地址

**函数成员的内存分布**

函数成员不占用堆内存，而是存放在`.text`段里。

**静态成员变量**

由所有对象共享，存储在全局区（或静态区）。

**this指针**

每个成员函数调用的时候，编译器会传递一个this指针，指向当前调用该函数的对象。this指针本质上是一个函数传递的参数。

静态成员没有this指针（标准一点来说就是没有办法被this指针管理），因为他存储在全局区，而不是在对象所在的堆内存中。this指针（一般就是对象的首地址）只能管理存储在对象堆内存中的数据。

# 12.静态变量

1. 初始化只执行一次，但可以多次赋值

2. **静态局部变量和静态全局变量**

   静态局部变量和静态全局变量都存储在“全局/静态存储区”，但各自的作用不同

   - 静态局部变量：延长生命周期，下一次访问值保留（但作用域不变）
   - 静态全局变量：为了限制访问范围在所在文件内

# 13.this指针

this 是一个指针，**指向调用当前成员函数的类对象的首地址**。只有非静态成员函数才能使用 this 指针，全局函数和静态成员函数无法使用 this，因为它们不属于类的具体对象。

隐含传递：在**调用类的<u>非静态成员函数</u>**时，编译器会自动将调用该函数的对象地址作为隐式参数传递给函数，即 this 指针。this 指针并不存储在对象中，因此它不会影响 sizeof(对象) 的结果。

编译器在类的成员函数中隐式添加了一个参数 `T *const this`，表示指向当前对象的指针。例如：

```cpp
int func(int p);
```

编译器实际上会将其转化为：

```cpp
int func(A *const this, int p);
```

当调用成员函数时，编译器会自动将对象的地址通过 `this` 指针传递给函数。因此，即使没有显式写出 `this`，编译器也会自动加上。

**this指针的创建时间**

-  this 指针在成员函数开始执行之前被创建，在成员函数执行结束后清除。
- 如果 class 或 struct 中没有方法，实际上并没有 this 指针存在。此时类对象只被当作一个普通的 C struct 使用。

**this指针的存放位置**

可以是栈、寄存器中或全局变量，取决于寄存器。

# 14.类的大小

- 空类

  ```cpp
  class A {}
  ```

  大小为1字节，每个实例需要一个独立的内存地址。

- 有虚函数的类

  ```cpp
  class A {
  	virtual void func() {}
  }
  ```

  由于每个包含虚函数的类都有一个虚函数表，所以创建出来的对象需要有一个指针指向该虚函数表（vptr）。

  - 32位系统：4字节（地址大小是32位，所以32/8=4）
  - 64位系统：8字节

- 类中有静态成员

  静态成员存放在全局区，不占对象内存空间

- 类中有普通成员变量

  需要考虑内存对齐

# 15.内存对齐及其原因

概念：内存对齐是指计算机在分配内存时，为了提高访问速度和内存利用率，**对数据的存储地址按照一定规则进行调整的过程**。对齐的目的是让 CPU 高效地访问内存。
