# 1.mac地址和ip地址

- ip地址（32位）：用于局域网之间的路由，ip地址指定的是设备所在的局域网，是门牌号
- mac地址（48位）：用于局域网内部的数据传输，每个网卡有自己独一无二的mac地址，是身份证

数据在小型的**交换机局域网**内传输，只需要MAC地址，在局域网之间跨网络传输，才需要ip地址。

# 2.数据拆解过程

数据包从里到外是：

1. **最内层：应用数据**（应用层）
2. **第 4 层：TCP 头**（传输层）
3. **第 3 层：IP 头**（网络层）
4. **第 2 层：MAC 帧头**（数据链路层）
5. **最外层：比特流**（物理层）

所以**数据包外面包的是二层（MAC），再里面是三层（IP），再里面是四层（TCP），最里面是应用数据**。

---

**1.电脑发出来 → 先到交换机**

交换机是**二层设备**，它只拆**最外面那层皮（二层 MAC 头）**：

- 只看**目的 MAC**
- 看完就转发，**不往里拆三层（IP）**

也就是：**交换机只剥到第 2 层（数据链路层）就停手。**

---

**2.交换机转发 → 到 路由器**

路由器是**三层设备**，它要：

1. 先把**最外面的二层 MAC 头剥掉**（拆到二层）
2. 再往里剥一层，拆到三层 IP 头
   - 看**目的 IP**
   - 查路由表
3. 然后**重新包一个新的二层头**，发往下一跳

也就是：**路由器剥到第 3 层（网络层）。**

# 3.网络拓扑

如果想要往外发，需要先发给该局域网的唯一的一个对外路由器（一个局域网对外的路由器只能有一个），数据在路由器中被封装（也就是说包含mac地址的数据被封成一个包裹，在外面再套一层写上ip地址），之后交给上层网络。

```bash
【你的局域网】
        ↓
PC1   PC2   手机   打印机  智能电视
 │     │     │      │       │
 └─────┴─────┴──────┴───────┘
              ↓
           【交换机】
              ↓
        （交换机某一端口）
              ↓
           【路由器】
              ↓
        光猫 / 上级网络 / 互联网
```

# 4.网段

通俗理解：网段是一个局域网中**多个分组**的编号，每个网段就是一个**独立的 IP 小组**

标志这个网络小组的可用ip地址的范围。

网段的规则为：

```
A.B.C.D/xx
```

前xx（通常是24，16，8）位表示网络地址，后32-xx位表示主机位。

**通用规则（所有网段都适用）**

- 一段里**第一个 IP** = 网络地址（全 0）→ **不能给设备用**

- 一段里**最后一个 IP** = 广播地址（全 1）→ **不能给设备用**

  > **广播**：不是发给**某一台设备**，而是发给**网段内所有设备**
  >
  > 设备一看到这个目标 IP，就知道：“这是广播，我也要听”

- 中间的才是**可用 IP**

---

**案例：/23可能包含/24的ip地址是吗**

/23 网段足够大，可以包含整整两个 /24 网段。

因为第24位或0或1。

# 5.特殊的ip地址

## 5.1 本机 / 本地专用

1. 0.0.0.0
   - 含义：**本机所有网卡、所有地址**
   - 工程用途：
     - 服务器监听（让服务在所有网卡都生效）
     - 路由表中代表**默认路由**
2. 127.0.0.1
   - 含义：**本机回环地址**
   - 作用：自己访问自己，不经过真实网卡

## 5.2 私网 IP（工程最常用）

### 1. A 类私网

- 10.0.0.0 ~ 10.255.255.255

  大型企业、园区网、运营商内网常用

### 2. B 类私网

- 172.16.0.0 ~ 172.31.255.255

  中型网络、部分企业 VPN

### 3. C 类私网

- 192.168.0.0 ~ 192.168.255.255

  家用 / 小型办公最常见

> 这三类都**不能在公网上直接路由**，只能内网用。

# 6.端口

- 区分：
  - 物理端口：网卡等接口
  - 虚拟端口：操作系统模拟出来的端口

## 6.1 进程和端口的关系

- 进程要上网通信 → **必须占用一个 / 多个端口**
- 一个进程可以**同时占多个端口**
- 一个端口**同一时间只能给一个进程用**（不能抢）

**举例：Web 服务**

- 进程占用 **80 端口**
- 来了 100 个用户请求
- 进程开 **100 个线程**分别处理
- **所有线程共用同一个 80 端口**

## 6.2 端口分类

### 6.2.1 知名端口

- 范围：**0~1023**
- 特点：
  - 系统、标准服务固定使用
  - **需要管理员权限才能绑定**
- 常用例子：
  - 20/21：FTP
  - 22：SSH
  - 23：Telnet
  - 25：SMTP
  - 53：DNS
  - 80：HTTP
  - 443：HTTPS
  - 3389：远程桌面

### 6.2.2 注册端口

- 范围：**1024 ~ 49151**

  许多应用和中间件常用，可能有约定但不一定是系统保留

- 特点：

  - 给软件、程序、服务**注册**使用
  - 普通用户权限也能用

- 常见例子：

  - 3306：MySQL
  - 5432：PostgreSQL
  - 6379：Redis
  - 8080：备用 HTTP

> 注册：把自己的信息报上去，让系统知道“我是谁、我在哪、怎么找到我”。
>
> 例如：用户服务启动后，会向“注册中心”报备：
>
> - 服务名：`user-service`
> - IP：`10.0.0.12`
> - 端口：`8080`
>
> 以后服务A想调用服务B时，就不用手写死 IP，而是去注册中心问：`serviceB` 现在在哪？
>
> 注册中心回答：在 `10.0.0.12:8080`

### 6.2.3 动态 / 私有 / 临时端口

- 范围：**49152 ~ 65535**

  一般给**客户端程序临时分配**，或者程序自己临时使用

- 特点：

  - 操作系统**自动分配**给客户端
  - 客户端发起连接时用
  - 不用于服务监听

- 比如：你浏览器打开网页，本地就用这类端口。

# 7.TCP-IP四层模型

## 7.1 应用层

**应用层协议规定应用程序之间通信的内容、格式和语义。**

它负责：

- 定义应用之间怎么交流
- 规定数据长什么样
- 规定请求和响应的格式

常见协议有：

- **HTTP / HTTPS**：规定网页请求、响应这些“应用数据怎么组织”（数据长什么样）
- **FTP**：规定文件上传、下载这些“应用数据怎么组织”（文件怎么传、怎么列目录）
- **SMTP**：规定发送邮件这些“应用数据怎么组织”（邮件内容、收件人、发件人怎么表示）
- **DNS**：规定域名查询这些“应用数据怎么组织”（域名和 IP 地址怎么对应、查询报文怎么写）
- **SSH**：规定远程登录、远程执行命令这些“应用数据怎么组织”（怎么认证、命令和返回结果怎么传）

## 7.2 传输层

**传输层规定数据如何在两个进程之间传输。**

它主要关心：

- 端口
- 分段
- 可靠传输还是尽力传输
- 顺序
- 重传
- 流量控制

最常见的协议：

- **TCP**：规定运输层数据如何可靠传输（如何分段、编号、确认、重传）
- **UDP**：规定运输层数据如何尽力而为地传输（如何加上端口信息后直接发送）

## 7.3 网络层

**网络层规定数据如何通过 IP 寻址和路由，到达目标主机所在的网络，并最终送到目标主机。**（<u>管跨网通信</u>）

网络层负责：

- 主机到主机的寻址
- 跨网络路由
- 决定数据包走哪条路

最核心的协议：

- **IP**
- 以及配套的 **ICMP**

## 7.4 网络接口层

**网络接口层负责把数据在当前链路上发送给下一跳设备。**（<u>管局域网内部通信</u>）

这一层负责：

- 在当前物理网络中传输数据
- 处理网卡、MAC 地址、以太网、Wi-Fi
- 把数据真正发到线上

它对应的常见技术有：

- **以太网 Ethernet**
- **Wi-Fi**
- **ARP**（常归到这一侧理解）
- 网卡驱动、交换机转发等

## 8.2 TCP协议

### 8.2.1 协议解释

![](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F92049a68b648b0ce7cd1ebabc8674396.png)

- 序号：这个报文段所携带的数据部分中，第一个字节在整个字节流中的编号（从0开始编号）。

- 确认号：期望收到对方下一个报文段的第一个数据字节的序号

  > 确认号由接收方填写，用来告诉发送方哪些字节已经收到
  >
  > 发送方发送数据时虽然不用自己填写这个确认号，但必须根据对方返回的确认号来决定后续怎么发。

- 首部长度：首部有多少个32位（上图中一行是32位，即多少行）

- 控制位：用来说明这个报文段的作用

  - **SYN**：同步标志，用于**建立连接**时同步序号

  - **ACK**：确认标志，表示**确认号字段有效**，仅当 ACK=1 时确认号字段才有效，TCP 规定在连接建立后所有传达的报文段都必须把 ACK 置1

  - **FIN**：终止标志，表示**发送方的数据已经发送完毕，请求释放连接**，当 FIN = 1 时，表明此报文段的发送方的数据已经发送完毕，并要求释放运输连接

  - **RST**：复位标志，表示**TCP 连接出现严重异常，需要立即重置连接** 

  - **PSH**：推送标志，表示**接收方应尽快将该报文段交付给应用进程**

    > 当两个应用进程进行交互式的通信时，有时在一端的应用进程希望在键入一个命令后立即就能够收到对方的响应。在这种情况下，TCP 就可以使用推送（push）操作，这时，发送方TCP 把 PSH 置 1，并立即创建一个报文段发送出去，接收方收到 PSH = 1 的报文段，就尽快地（即“推送”向前）交付给接收应用进程，而不再等到整个缓存都填满后再向上交付

  - **URG**：紧急标志，此位置 1 ，表明紧急指针字段有效，它告诉系统此报文段中有紧急数据，应尽快传送

- 窗口：指发送本报文段的一方的接收窗口（而不是自己的发送窗口）

- 校验和：校验和字段检验的范围包括首部和数据两部分，在计算校验和时需要加上 12 字节的伪头部

- 紧急指针：仅在 URG = 1 时才有意义，它指出本报文段中的紧急数据的字节数（紧急数据结束后就是普通数据），即指出了紧急数据的末尾在报文中的位置，注意：即使窗口为零时也可发送紧急数据

- 选项：长度可变，最长可达 40 字节，当没有使用选项时，TCP 首部长度是 20 字节

### 8.2.2 发送流程

TCP是可靠传输，每次发送数据之后都要确认数据是否收到，即：**在有效时间内是否能够收到接收方的ACK应答**。

- 如果超过时间还没收到接收方的ACK应答，则发送方重传

![image-20260327175713303](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F98ae3344a57c7af1aee2887ff77b9f7b.png)

上述这种是理论上的发送流程，实际使用时，接收方每次发送一个数据都要等待应答，速度太慢。

---

**tcp流量控制**

比较容易想到，发送方可以**一次发送一批数据**给接收方。

然而，问题又来了，一次应该发送多少数据给接收方，接收方才能来得及接收呢？这就是tcp流量控制。

![image-20260327180310208](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2Ffc12a36fc9fd87f609f151b5d4997dae.png)

每次发送方在发送数据之前，接收方都要告诉发送方：目前发送方的窗口大小（目前还可以发送多少未确认字节的数据），因为每次接收方能够接收的数据大小是变化的，有可能这一次多，下一次就少了。

而这个**窗口的大小就写在了每次接收方的ACK应答中**。

即滑动窗口规定了：**发送方当前最多能有多少“已发送但未确认”的数据在路上**

> **发送方每次收到 ACK 后，会更新自己对“当前还能发多少数据”的判断，然后决定接下来还能继续发送多少。**
>
> 假设当前允许窗口是 **8000 字节**。
>
> 发送方现在已经发出去 **6000 字节**，其中只有 **3000 字节**被确认了。
>
> 那当前未确认的数据就是：
>
> - 6000 - 3000 = **3000 字节**
>
> 所以理论上还能再发：
>
> - 8000 - 3000 = **5000 字节**

![image-20260327180804357](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2Fb0cfd40fe4d2e842849075abd0e9df65.png)

**数据分块**

- 在连接建立的时候，发送方首次发送SYN请求建立连接；

- 接收方返回SYN+ACK响应请求，在返回的应答中，包含了当前接收方能够承受的窗口是3000字节（一个3000字节的缓冲区），且由于**中间网络MTU的限制**（一般为1500字节），接收方规定发送方一次TCP数据包最大的有效负载为1000字节，即MSS。

  > MTU=IP首部（一般20字节）+TCP首部（一般20字节）+有效负载
  >
  > 所以，在MSS=1000的情况下，MTU=1000+20+20=1040

- 在MSS=1000的情况下，发送方将总的6000字节的数据拆成一份1000字节，分块发送三份数据过去

![image-20260327182324388](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F10b8468a05cc43d7b62853bb56ddc9a7.png)

**窗口变化过程**

如果接收方处理数据的速度很慢，在回复ACK的时候，将窗口设置为0

![image-20260327183650839](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F2332496b221e754d0b2f013e96fe8842.png)

过了一会儿，接收方处理完了2000字节的数据，缓冲区中还剩下1000字节没有处理。

此时，需要将窗口值设置为2000字节，**可以再次发送确认告知新的窗口值**。

发送方接着发送2000字节的数据给接收方，再次将3000字节的窗口填满

![image-20260327183844178](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F8d644153b8b77f4dcf1805ab65077153.png)

此时，接收方还没来得及处理数据，只能给发送方返回窗口大小为0字节

过了一会儿，接收方处理完2000字节的数据（此时剩余1000字节），又因为其他的原因，发现原本3000字节的窗口太大了，来不及处理，就收缩窗口，ACK告知窗口大小为2000字节。

接收方把剩下来的最后1000字节的数据发给接收方。

![image-20260327184354437](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2Ff663f43ac1b680233e2de41a9843814c.png)

**结束连接**

- 发送方发现数据发送完毕，就向接收方发送FIN关闭连接请求
- 接收方发送ACK确认表示收到请求
- 等到接收方数据处理完毕并清空了缓冲区，发送FIN来结束连接
- 发送发收到后发送ACK确认

**累计确认**

TCP 不是直接说“第几个包丢了”，而是通过“我现在连续收到哪儿了”来间接告诉发送方哪里丢了。

假设发送方连续发送 3 段数据：

- 段1：序列号 `1001`，长度 `1000`
- 段2：序列号 `2001`，长度 `1000`
- 段3：序列号 `3001`，长度 `1000`

意思就是：

- 段1携带字节 `1001 ~ 2000`
- 段2携带字节 `2001 ~ 3000`
- 段3携带字节 `3001 ~ 4000`

如果段1收到了，接收方会回：

- `ACK = 2001`

意思是：`2001` 之前的字节我都连续收到了，下一步请从 `2001` 开始发。

这叫 **累计确认**。

此时，如果：

- 段1：`1001~2000` 已到，且已经确认过，所以回过 `ACK=2001`

- 段2：`2001~3000` 丢了
- 段3：`3001~4000` 到了，但因为中间缺段2，所以只能一直回 `ACK=2001`

此时，ACK=2001，已接收但未确认的段3：3001~4000，发送方重传段2，接收方接收到段2后，连着段3一起确认，返回ACK=4001。

## 8.3 IP协议

![image-20211113205020978](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2F326622c52a244e8334e8fbc43a59d090.png)

- 版本号：IP 协议的版本。通信双方使用过的 IP 协议的版本必须一致，目前最广泛使用的 IP 协议版本号为 4（即IPv4)
- 头部长度：单位是 32 位（4 字节）
- 服务类型：一般不适用，取值为 0
- 总长度：指首部加上数据的总长度，单位为字节
- 标识（identification）：IP 软件在存储器中维持一个计数器，每产生一个数据报，计数器就加 1，并将此值赋给标识字段
- 标志（flag）：目前只有两位有意义
  - 标志字段中的最低位记为 MF。MF = 1 即表示后面“还有分片”的数据报。MF = 0 表示这已是若干数据报片中的最后一个
  - 标志字段中间的一位记为 DF，意思是“不能分片”，只有当 DF = 0 时才允许分片
- 片偏移：指出较长的分组在分片后，某片在源分组中的相对位置，也就是说，相对于用户数据段的起点，该片从何处开始。片偏移以 8 字节为偏移单位
- 生存时间：TTL，表明是数据报在网络中的寿命，即为`跳数限制`，由发出数据报的源点设置这个字段。路由器在转发数据之前就把 TTL 值减一，当 TTL 值减为零时，就丢弃这个数据报，常见为**64和128**
- 协议：指出此数据报携带的数据时使用何种协议，以便使目的主机的 IP 层知道应将数据部分上交给哪个处理过程，常用的 ICMP(1)，IGMP(2)，TCP(6)，UDP(17)，IPv6（41）
- 首部校验和：只校验数据报的首部，不包括数据部分
- 源地址：发送方 IP 地址
- 目的地址：接收方 IP 地址

## 8.4 ARP协议

![image-20211116222942537](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F03%2Fdee8cd0a89f5dc70ecedc4cdba408608.png)

- 硬件类型：1 表示 MAC 地址
- 协议类型：0x800 表示 IP 地址
- 硬件地址长度：6
- 协议地址长度：4 
- 操作
  - 1 表示 ARP 请求
  - 2 表示 ARP 应答
  - 3 表示 RARP 请求
  - 4 表示 RARP 应答

当一台主机要在某个局域网上把 IP 包交给“下一跳设备”时，如果还不知道对方的 MAC 地址，就会在这个局域网里发 ARP 请求，通过 IP 找到对应的 MAC。

这个思路在“**最后一跳**”上基本就是这样：

- 路由器把 IP 包转发到目标局域网
- 它发现目标 IP 就在自己直连的这个网段里
- 但它还不知道目标主机的 MAC
- 于是它在这个局域网里发 ARP（**广播**）：
  - “谁是这个 IP？”
- 目标主机回复自己的 MAC
- 路由器再把 IP 包封装成以太网帧发过去
