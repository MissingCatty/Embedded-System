# 1.概念

TCP通信是全双工的，两端都可以同时收发数据。

如果把一端的接收/发送功能关了，此时这一端只能发送/接收，这就叫半关闭状态。

# 2.代码

```c
#include <sys/socket.h>

int shutdown(int sockfd, int how);
/*
    - 功能：终止套接字的全部或部分连接（实现 TCP 半关闭）
    - sockfd：需要关闭的套接字文件描述符（connect_fd不是listen_fd）
    - how：断开方式，有三种取值：
        - SHUT_RD (0)：断开读。不再接收数据，丢弃接收缓冲区现有的货。
        - SHUT_WR (1)：断开写。不再发送数据，发送缓冲区剩下的货发完后，发送 FIN 包。
        - SHUT_RDWR (2)：同时断开读和写，相当于先调 SHUT_RD 再调 SHUT_WR。
    - 返回值：
        - 成功：0
        - 失败：-1，并设置 errno
*/
```

**shutdown和close有啥区别？**

想象一个场景：你要给服务器上传一个超大文件。

1. 你发送完文件后，你想告诉服务器：“我的货发完了（发送 `FIN`）”。

2. 但是，你还想等服务器处理完之后给你回一个“上传成功”的确认消息。

3. **如果你用 `close`**：你的读写全断了，服务器发回来的消息你收不到。

   > 假设你 `fork()` 了一个子进程，现在父子进程都有一个 `fd = 5` 指向同一个连接。
   >
   > **如果你调用 `close(5)`**：
   >
   > - 结果：父进程的 `fd` 没了，但子进程的 `fd` 还在。
   > - **TCP 连接不会断开**，因为引用计数还没到 0。
   >
   > **如果你调用 `shutdown(5, SHUT_RDWR)`**：
   >
   > - 结果：**瞬间爆炸**。
   > - **TCP 连接立刻发起断开序列（发 FIN）**。
   > - 即使子进程手里还握着那个 `fd`，它也发不出任何数据了，因为底层的“水管”已经被你物理切断了。

4. **如果你用 `shutdown(fd, SHUT_WR)`**：你发出了 `FIN` 告诉对方发完了，但你的读通道还开着，依然可以坐等服务器的回执。

   > - 发送`FIN`的本质是：告诉对方“我没话说了”（关闭自己的发送端）
   > - 接收`FIN`的本质是：收到对方的分手请求了（关闭自己的接收端）
