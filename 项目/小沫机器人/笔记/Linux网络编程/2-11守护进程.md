# 1.进程组

你平时在终端里敲一条命令，很多时候背后不一定只有一个进程。比如一条管道命令：

```bash
sleep 100 | cat
```

这里有两个进程，但对 shell 来说，它们属于同一个作业，所以 shell 会把它们放进同一个进程组里。

**同一时刻， 前台通常只有一个进程组，而后台可以有多个进程组。**

这样一来：

- 前台运行时，终端把输入交给这一组
- 你按 `Ctrl+C`，信号发给这一组（前台进程组里所有进程都会收到）
- 你按 `Ctrl+Z`，暂停这一组
- 放后台时，也是整组一起放

所以你可以记一句：**进程组是 shell 控制作业的基本单位。**

每个进程组都有一个编号，叫 **PGID**，也就是进程组 ID。一个进程属于哪个组，就看它的 PGID 是多少。**PGID 相同的进程，属于同一个进程组。**

---

**首进程**

一个进程组总要有人最先把这个组建起来。那个最先创建该组的进程，就叫 **进程组首进程**。

通常它有一个特点：**它自己的 PID = 这个进程组的 PGID**。

---

**新进程怎么进入进程组**

一般来说，一个新进程是通过 `fork` 从父进程创建出来的。默认情况下，**子进程会继承父进程的进程组 ID**。

也就是说：

- 父进程在哪个组
- 子进程刚生出来时通常也在哪个组

# 2.会话

会话 = 一组进程组的集合。

你可以把会话理解成：**一次终端交互环境的总容器**。

比如你打开一个终端，里面跑着一个 shell，这整个终端里的前台作业、后台作业，通常都在同一个会话里。

创建新会话的那个进程，叫 **会话首进程**。

它有两个特点：

- 它是这个会话的创建者
- 它的 **PID 会成为 SID（会话 ID）**

**一个会话中的所有进程，共享同一个控制终端**。这样 shell 才能在同一个终端里管理前台/后台作业。

- 新会话刚创建时，不一定立刻有控制终端
- 当这个会话的首进程第一次打开某个终端设备
- 这个终端就会成为该会话的 **控制终端**

一个会话里有前台组和后台组，在任意时刻，一个会话里的多个进程组中：

- **有且只有一个** 是终端的 **前台进程组**
- 其他的都是 **后台进程组**

**只有前台组能读终端输入**，系统规定：**只有前台进程组中的进程，才能从控制终端读取输入。**

因为终端一次只能明确服务一个“当前交互对象”。

```
控制终端
   |
   v
一个会话（SID）
   |
   +-- 前台进程组（唯一，能读终端输入，接收 Ctrl+C / Ctrl+Z）
   |
   +-- 后台进程组1
   |
   +-- 后台进程组2
   |
   +-- ...
```

# 3.案例

在这条命令里：

```bash
ls | grep a
```

如果你是**直接在终端里执行**它，没有加 `&`，那它整体就是一个**前台作业**。
 因此：

- `ls` 是前台进程组里的一个进程
- `grep a` 也是前台进程组里的一个进程

也就是说，**它们通常都属于同一个前台进程组**，不是一个前台一个后台。

如果你写成：

```bash
ls | grep a &
```

这时就不一样了。
 这整个管道作业会被放到**后台**。

于是：

- `ls` 在后台进程组里
- `grep a` 也在后台进程组里

# 4.进程组、会话操作函数

## 4.1 getpgrp：获取当前进程的pgid

```c
#include <unistd.h>
#include <sys/types.h>

pid_t getpgrp(void);
/*
	- 返回值：
		- 成功：返回当前进程所属的进程组 ID（PGID）
		- 失败，一般不会失败
*/
```

## 4.2 getpgid：获取指定进程的pgid

```c
#include <unistd.h>
#include <sys/types.h>

pid_t getpgid(pid_t pid);
/*
	- pid：
		- pid > 0：获取指定进程的进程组 ID
		- pid == 0：获取当前进程的进程组 ID
	- 返回值
		- 成功：返回进程组 ID（PGID）
		- 失败：返回 -1，并设置 errno
*/
```

## 4.3 setpgid：设置指定进程的pgid

```c
#include <unistd.h>
#include <sys/types.h>

int setpgid(pid_t pid, pid_t pgid);
/*
	- pid：要设置进程组的进程 ID
		- pid == 0 表示当前进程
	- pgid：目标进程组 ID
		- pgid == 0 表示使用 pid 指定进程的 PID 作为 PGID
	- 返回值：
		- 成功：返回 0
		- 失败：返回 -1，并设置 errno
*/
```

## 4.4 getsid：获取会话id

```c
#include <unistd.h>
#include <sys/types.h>

pid_t getsid(pid_t pid);
/*
	- pid：
		- pid > 0：获取指定进程的会话 ID
		- pid == 0：获取当前进程的会话 ID
	- 返回值：
		- 成功：返回会话 ID（SID）
		- 失败：返回 -1，并设置 errno
*/
```

## 4.5 setsid：创建会话

```c
#include <unistd.h>
#include <sys/types.h>

pid_t setsid(void);
/*
	- 参数：
		- 无
	- 返回值：
		- 成功：返回新会话的会话 ID（SID）
		- 失败：返回 -1，并设置 errno
*/
```

# 5.守护进程

## 5.1 概念

守护进程（**Daemon Process**）就是 **Linux 里的后台服务进程**。

它和普通进程最大的区别是：

- 它不是为了和用户直接交互
- 它通常在后台一直运行
- 专门负责长期提供某种系统服务，或者周期性处理任务

比如：

- 网络服务
- 日志服务
- 定时任务服务
- Web 服务

很多守护进程名字最后会带一个 **d**，比如：

- `httpd`
- `sshd`
- `inetd`

这个 `d` 基本就表示 daemon。

---

守护进程没有控制终端，服务器程序本质上就需要：

- 长时间运行
- 不依赖某个用户终端
- 随时响应请求
- 用户退出登录后也不能停

这正好就是守护进程的工作方式。

所以很多 Linux 服务本质上都是守护进程。

## 5.2 守护进程是怎么来的

守护进程通常是**普通进程 fork 出子进程后，子进程再通过 `setsid()` 脱离原控制终端和会话，最后独立在后台长期运行**形成的。

也就是`setsid`的作用是，让当前进程脱离原来的进程组和会话，创建一个新的会话，**并成为这个新会话的首进程**。

## 5.3 为什么一定要守护进程

子进程被创建之后，跟着销毁父进程，不是也能实现进程一直在后台运行吗？

原因是：**不脱离原会话，它仍然可能受原来的终端和 shell 影响**。

- 仍然属于原会话
- 仍然和原控制终端有关系
- 关闭终端时，可能收到相关信号
- shell 退出时，可能影响它
- 读写终端时，可能受到作业控制限制

所以它只是**后台作业**，不等于**守护进程**。

## 5.4 案例

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/stat.h>
#include <time.h>
#include <fcntl.h>

int main(void)
{
    pid_t pid = fork();

    if (pid == -1)
    {
        perror("fork error");
        exit(1);
    }
    else if (pid > 0)
    {
        printf("parent process exit.\n");
        exit(0);
    }

    /* 1. 脱离原会话 */
    pid_t sid = setsid();
    if (sid < 0)
    {
        perror("setsid error");
        exit(1);
    }

    /* 2. 切换工作目录，避免占用挂载点 */
    chdir("/");

    /* 3. 重设文件权限掩码 */
    umask(0);

    /* 4. 关闭标准输入/输出/错误 */
    close(STDIN_FILENO);
    close(STDOUT_FILENO);
    close(STDERR_FILENO);

    while(1) {
        FILE *fp = fopen("/tmp/mydaemon.log", "a");
        if (fp != NULL) {
            time_t t = time(NULL);
            fprintf(fp, "daemon is running: %s", ctime(&t));
            fclose(fp);
        }
        sleep(5);
    }
}
```

注意看这个程序里的2，3，4

### 2：为什么要切换工作目录，避免占用挂载点

因为有些守护进程起来之后就不需要其加载进程所在的工作目录了，可以把其所在的文件系统卸载了。

比如你在 U 盘里有个程序：

```bash
/mnt/usb/myprog
```

启动后，程序代码已经装入内存，后面它主要就是：

- 做计算
- 处理网络请求
- 定时执行任务

**如果它不再去访问 `/mnt/usb` 里的别的文件，那它其实没必要还把当前工作目录留在 `/mnt/usb`**。此时U盘就可拔了，但如果不切换工作目录，U盘这个文件系统就卸载不掉。

### 3：为什么要重设文件权限掩码

守护进程是从别的进程继承来的，它继承到的 `umask` 可能是创建者环境留下的，不一定适合它自己。如果不重设，守护进程以后创建日志文件、套接字文件之类时，权限可能莫名其妙变窄。

比如：本来想创建 `666` 权限的文件，结果因为 `umask` 的存在，实际变成别的权限。

不过实际开发里不一定非要设成 `0`，也可以设成更合适的值，比如 `022`。教材里常写 `umask(0)`，是为了先把旧影响清掉。

### 4：为什么要关闭标准输入/输出/错误

因为守护进程要**脱离终端独立运行**。

标准输入、标准输出、标准错误这三个文件描述符通常默认都和终端相关：

- `stdin`：从终端读输入
- `stdout`：往终端打印
- `stderr`：往终端输出错误

如果不关掉，就说明它还可能和原来的终端保持输入输出关系，这和“守护进程不依赖终端”这个目标不符。关闭它们的好处是：

- 避免继续读写原终端
- 避免终端关闭后带来问题
- 避免后台进程把内容乱输出到终端

很多时候关闭后还会进一步把它们重定向到：

- `/dev/null`（一个特殊的设备文件，往里写的东西都会被丢掉）
- **日志文件**

这样更规范。
