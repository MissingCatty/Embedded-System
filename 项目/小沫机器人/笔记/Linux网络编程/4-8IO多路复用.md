# 1.概念

I/O 多路复用允许一个进程**同时监听多个文件描述符**（fd），一旦某个 fd 就绪（可读/可写），内核就通知程序。

**使用场景**：高并发网络服务器

- **传统模型：** 一个连接对应一个线程。当连接数达到 10,000 时，系统会因为线程切换带来的开销（上下文切换）而崩溃。

- **多路复用：** **只需要一个（或极少数）线程**就能同时监控 10,000 个连接。只有当某个客户端真的发来数据时，程序才会去处理。

---

**传统IO方式有两种模型：阻塞/非阻塞式**

**阻塞式**

每个连接线程里，调用`recv`函数接收数据，如果没有数据，就一直阻塞在这。

- 优点：（对CPU来说）不占用CPU资源
- 缺点：（对线程本身来说）无数据时一直阻塞，不能做其他的事

**非阻塞式**

调用`recv`函数读取失败的时候，直接返回，相当于轮询。

- 优点：（对线程本身来说）线程读取不到时可以继续做其他事
- 缺点：（对CPU来说）会消耗CPU资源，由于一直在运行，CPU需要频繁的切换到该线程，线程调度增加了开销

---

**IO多路复用**

上述两种方式在高并发场景下基本都不可行，因为基本都是一个线程管一个连接。

> 因为这两种方式不可能只用一个线程实现。
>
> - 阻塞式：一直等待一个连接的数据，后续连接没法查看
> - 非阻塞式：一直轮询，如果有10000个，轮询开销太大，速度也慢

一个线程管一个连接太奢侈了，有多个连接的时候往往要开多个线程，**每个线程都要开辟独立的栈空间、内核调度开销、创建和销毁都有成本**。同时大量的线程会涉及大量的线程切换。

因此，**在高并发场景下，往往使用一个线程去管理所有的连接。**

具体的管理方式有三种：select，poll，epoll

## 2.1 select

### 2.1.1 基本功能

**只会通知有几个任务可用，但不知道具体哪几个任务**，还需遍历（与NIO模型略有不同）

底层通过二进制位图来表示哪些cfd有数据到达。

### 2.1.2 使用流程

1. 首先需要构造一个文件描述符列表，里面存放需要被管理的文件描述符

2. 调用一个系统函数，去管理这个列表中的文件描述符

   > 该系统函数是阻塞的，只有当列表中的文件描述符有一个或多个进行IO操作时，才返回

3. 在返回时，会告诉进程有多少描述符要进行IO操作

### 2.1.3 相关函数

在使用 `select` 函数之前和之后，必须通过以下宏来处理 `fd_set`（文件描述符集合），因为 `fd_set` 本质上是一个位图（bitmap）。

```c
#include <sys/select.h>

// 创建文件描述符集合
fd_set readfds;
/*
    - 类型本质：fd_set 是一个结构体，内部封装了一个长整型数组，表现为一个“位图”（bitmap）。
    - 存储位置：通常定义在栈上（局部变量）。
    - 限制：其大小由内核宏 FD_SETSIZE 定义，通常为 1024。
    - 初始化要求：刚创建的 fd_set 包含随机的垃圾值，使用前必须用 FD_ZERO 初始化。
*/

void FD_ZERO(fd_set *set);
void FD_SET(int fd, fd_set *set);
void FD_CLR(int fd, fd_set *set);
int  FD_ISSET(int fd, fd_set *set);

/*
    - 功能：操作 fd_set 集合
    - 参数说明：
        - fd：需要操作的文件描述符
        - set：fd_set 类型的集合
    - 具体宏的功能：
        - FD_ZERO：清空集合，将位图所有位设为 0（初始化时必用）
        - FD_SET：将指定的 fd 加入集合，位图中对应位置 1
        - FD_CLR：将指定的 fd 从集合中删除，位图中对应位置 0
        - FD_ISSET：判断 fd 对应的位是否为 1
    - 返回值：
        - FD_ISSET：若 fd 在集合中返回非 0（真），否则返回 0（假）
        - 其他宏：无返回值
*/
```

接下来是select函数：

```c
#include <sys/select.h>
#include <sys/time.h>
#include <sys/types.h>
#include <unistd.h>

int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);
/*
    - 功能：委托内核监控一组文件描述符的读、写、异常事件。
    - nfds：监控的所有文件描述符中最大的那个 fd + 1（内核遍历的范围）。
    - readfds：要监控“可读”事件的 fd 集合。
    - writefds：要监控“可写”事件的 fd 集合（通常传 NULL）。
    - exceptfds：要监控“异常”事件的 fd 集合（通常传 NULL）。
    - timeout：超时时间：
        - NULL：永久阻塞，直到有 fd 就绪。
        - 0：立即返回（轮询）。
        - struct timeval：等待指定的时间。
        struct timeval { 
    		long tv_sec; // seconds
    		long tv_usec; // microseconds
		};
    - 返回值：
        - > 0：就绪的文件描述符总数。
        - 0：超时，没有 fd 就绪。
        - -1：执行失败，并设置 errno。
*/
```

![image-20211124233108458](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F04%2F5a3b736d3f1d189c894cb59eeae3d465.png)

**！！！需要注意**：输入的`readfds`在调用`select`后会被改掉，因为**我们理解的readfds的作用和select函数理解的不一样**，使用者认为哪些要管理，`select`认为哪些有数据到达。

> 可以把 `select` 想成一个“筛子”。
>
> 你先给它一整筐：
>
> ```
> allfds = {listen_fd, cfd1, cfd2, cfd3}
> ```
>
> 然后你说：
>
> > 帮我筛一下，哪些现在可读？
>
> 它返回给你的那一份，就是筛出来的结果：
>
> ```
> tmpfds = {cfd2, cfd3}
> ```
>
> 但原来那整筐你自己得留着，不能只剩筛出来这点。

### 2.1.4 可能存在的问题

**问题一**

细心的我发现一个问题：这里的`FD_SET`的大小由内核确定，存储的fd范围是`0-1023`，那如果一个进程文件描述符表的范围是`0~65535`，那有可能被分配得到的cfd超过了`FD_SET`的范围，导致溢出错误。

**解决方法**

- 方法 A：改用 poll

  `poll` 抛弃了位图的概念，改用一个**结构体数组**。你的进程描述符表有多大，你就可以开多大的数组，完全同步。

- 方法 B：改用 epoll（推荐）

  `epoll` 在内核中维护监听列表，它不仅没有 1024 的限制，而且效率不会随着连接数增加而线性下降。

**问题二**

上图中，由于管理过程需要交给内核，而`FD_SET`又是由用户进程创建的，涉及内核态和用户态的切换。

- **调用时（用户态 -> 内核态）：** 你通过 `FD_SET` 准备好的 `readfds` 等集合，在调用 `select` 系统调用时，会被完整地**从用户空间拷贝到内核空间**。内核需要这份名单来知道该盯着哪些 socket。

- **返回时（内核态 -> 用户态）：** 内核监控完毕后，会修改这份位图（只保留有数据的 fd 位），然后将修改后的位图**从内核空间重新拷贝回用户空间**，并唤醒阻塞的进程。

> - **既然状态的切换十分耗时，为什么不设计成直接在fd中创建`FD_SET`？**
>   - `select` 诞生于 20 世纪 80 年代（BSD 4.2）。当时的设计哲学倾向于**简单、无状态**（因为在 `select` 设计的年代，服务器内存可能只有几兆字节）
>
> - **`poll`函数解决了问题一，但问题二还是没解决**
>   - `poll` 解决了 1024 的数量限制，但它依然要求用户每次调用时把“结构体数组”传给内核。它依然是**无状态**的。
>
> - **`epoll`函数才是最终形态**

### 2.1.5 代码实现

实现目标：客户端A,B,C,D连接到服务器分别对应文件描述符3,4,100,101

```c
#include <sys/select.h>
#include <sys/time.h>
#include <sys/types.h>
#include <unistd.h>
#include <sys/socket.h>
#include <stdio.h>
#include <arpa/inet.h>
#include <string.h>  // 【修改1】新增头文件，因为下面用到了 memset

#define SERVER_IP "172.16.0.6"
#define SERVER_PORT 9999

int main(void)
{
    /* 1. 创建socket */
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0)
    {
        perror("create listen fd error");
        return 1;
    }

    /* 2. 绑定端口 */
    int opt = 1;
    setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)); // 【修改2】新增：设置端口复用（放在 bind 前）

    struct sockaddr_in saddr;
    memset(&saddr, 0, sizeof(saddr));  // 【修改3】新增：结构体先清零
    saddr.sin_family = AF_INET;
    inet_pton(AF_INET, SERVER_IP, &saddr.sin_addr);
    saddr.sin_port = htons(SERVER_PORT);

    if (bind(listen_fd, (struct sockaddr *)&saddr, sizeof(saddr)) == -1)
    {
        perror("socket bind error");
        close(listen_fd);  // 【修改4】新增：出错前关闭 listen_fd
        return 1;
    }

    /* 3.开始监听 */
    if (listen(listen_fd, 10) == -1)
    {
        perror("socket listen error");
        close(listen_fd);  // 【修改5】新增：出错前关闭 listen_fd
        return 1;
    }

    /* 4. 创建fd_set */
    fd_set readfds, tmpfds;   // 【修改6】新增 tmpfds：readfds 管理全部 fd，tmpfds 给 select 使用
    FD_ZERO(&readfds);
    FD_SET(listen_fd, &readfds); // listen_fd也要被管理
    int max_fd = listen_fd;

    /* 5. 等待连接 */
    printf("服务器正在等待被连接...\n");
    while (1)
    {
        tmpfds = readfds;  // 【修改7】关键修改：每次 select 前拷贝一份，避免 readfds 被 select 改坏

        /* 6. 调用select等待被管理的fd有数据到达 */
        int ret = select(max_fd + 1, &tmpfds, NULL, NULL, NULL);  // 【修改8】保存 select 返回值
        if (ret < 0)
        {
            perror("select function error");
            close(listen_fd);  // 【修改9】出错前关闭 listen_fd
            return 1;
        }

        // 【修改10】删除 ret == 0 分支
        // 因为这里 timeout 传的是 NULL，select 会一直阻塞，不会超时返回 0

        // >0 说明检测到有fd对应的缓冲区中数据有了变化
        if (FD_ISSET(listen_fd, &tmpfds))  // 【修改11】这里检查的是 tmpfds，不是 readfds
        {
            // 是listen_fd有变化
            /* 7. 建立连接 */
            struct sockaddr_in client_addr;
            memset(&client_addr, 0, sizeof(client_addr));  // 【修改12】修正：清零 client_addr，不是清 saddr
            socklen_t addrlen = sizeof(client_addr);       // 【修改13】修正：使用 socklen_t
            int cfd = accept(listen_fd, (struct sockaddr *)&client_addr, &addrlen);

            if (cfd < 0)   // 【修改14】新增：accept 失败时不能继续往下执行
            {
                perror("accept error");
            }
            else
            {
                char client_ip[16];
                printf("[连接成功]，客户端ip：%s:%d\n",
                       inet_ntop(AF_INET, &client_addr.sin_addr, client_ip, sizeof(client_ip)), // 【修改15】更标准写法：传 &client_addr.sin_addr
                       ntohs(client_addr.sin_port));

                /* 8. 更改FD_SET */
                FD_SET(cfd, &readfds);   // 【修改16】新连接加入“长期管理集合”
                max_fd = cfd > max_fd ? cfd : max_fd;
            }
        }

        // 剩下的轮询
        for (int i = 0; i <= max_fd; i++)
        {
            if (i != listen_fd && FD_ISSET(i, &tmpfds))  // 【修改17】只处理 tmpfds 中“本轮就绪”的 fd；并跳过 listen_fd
            {
                char recv_buf[1024];
                int len = recv(i, recv_buf, sizeof(recv_buf), 0);

                if (len < 0)
                {
                    perror("recv error");

                    close(i);              // 【修改18】修正：某个客户端出错，只关闭这个客户端，不让整个服务器退出
                    FD_CLR(i, &readfds);   // 【修改19】从长期管理集合中移除

                    // 【修改20】如果关闭的是当前最大 fd，就重新计算 max_fd
                    if (i == max_fd)
                    {
                        for (int j = max_fd; j >= 0; j--)
                        {
                            if (FD_ISSET(j, &readfds))
                            {
                                max_fd = j;
                                break;
                            }
                        }
                    }
                }
                else if (len == 0)
                {
                    printf("fd%d连接关闭\n", i);
                    close(i);
                    FD_CLR(i, &readfds);   // 【修改21】客户端正常关闭时，也要从长期管理集合里删除

                    // 【修改22】重新寻找当前最大的 fd
                    if (i == max_fd)
                    {
                        for (int j = max_fd; j >= 0; j--)
                        {
                            if (FD_ISSET(j, &readfds))
                            {
                                max_fd = j;
                                break;
                            }
                        }
                    }
                }
                else
                {
                    printf("fd%d接收：", i);
                    fflush(stdout);
                    write(1, recv_buf, len);
                }
            }
        }
    }

    close(listen_fd);
    return 0;
}
```

## 2.2 poll

```c
#include <poll.h>

int poll(struct pollfd *fds, nfds_t nfds, int timeout);

/*
    - 功能：委托内核监控一组文件描述符的状态。
    - fds：一个 struct pollfd 类型的数组，存放需要监控的 fd 及其事件。
    - nfds：fds 数组中有效元素的个数。
    - timeout：超时时间（毫秒 ms）：
        - -1：永久阻塞，直到有 fd 就绪。
        - 0：立即返回（非阻塞轮询）。
        - >0：等待指定的毫秒数。
    - 返回值：
        - > 0：就绪的文件描述符总数。
        - 0：超时。
        - -1：失败，并设置 errno。
*/
```

```c
struct pollfd {
    int   fd;         /* 委托内核监控的文件描述符 */
    short events;     /* 委托内核监控的事件（输入参数） */
    short revents;    /* 文件描述符实际发生的事件（输出参数，由内核填充） */
};

/*
	常用的事件宏（Events/Revents）：
		- POLLIN：数据可读。
		- POLLOUT：数据可写。
		- POLLERR：发生错误。
*/
```

使用流程：

1. **用户态准备：** 你创建一个 `struct pollfd` 数组，填入你想监控的 `fd` 和 `events`（比如 `POLLIN`）。
2. **内核监控：** 调用 `poll` 后，内核拷贝这个数组到内核态，并进行线性遍历。
3. **内核回填：** 如果某个 fd 有事件发生，内核直接修改该元素中的 `revents` 成员，而**不会破坏**你原始设置的 `events`，也不会删除数组中的`fd`。
4. **用户判断：** `poll` 返回后，你依然需要 O(n) 遍历数组，用 `if(fds[i].revents & POLLIN)` 来判断哪个 fd 真正就绪了。

---

使用poll可以解决`select`的两个问题：

- 最大是1024
- 输入的位图会被`select`函数修改（本质是位图中一位表示不了多维信息，比如：该fd是否需要管理和是否有数据要读）

## 2.3 epoll

会通知具体有哪几个任务可用

### 2.3.1 epoll使用流程

```c
int lfd = socket();
bind();
listen();
int cfd = accept();
int epfd = epoll_create(2000);
```

上述过程中的

```c
int epfd = epoll_create(2000);
```

在内核中创建一个数据结构，类型为`eventpoll`，返回值是一个文件描述符，用户程序可以**通过该文件描述符对内核中的数据结构进行操作**。

![image-20211127170241104](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/2026%2F04%2Fdb01f06dec36d3f84964b8ee69da70d6.png)

上图中，`eventpoll`里有两个重要的数据结构，

- `rbr`：用于**检索**是否有想要的`fd`并找到对应的项，节点来自每个 `epitem` 里的 `rdllink` 成员

- `rdlist`：就绪列表，节点和`rbr`一致，**表示有哪些`fd`已经就绪**

  ```c
  eventpoll.rdllist  --->  epitem.rdllink  --->  epitem.rdllink  ---> ...
  ```

对内核内存里的`eventpoll`操作的接口如下：

- epoll_ctl：操作函数，可以指定多种操作方式（更改红黑树`rbr`的节点，增删改查）
- epoll_wait：返回已经就绪的fd（返回`rdlist`就绪列表节点）

### 2.3.2 函数使用

```c
#include <sys/epoll.h>

int epoll_create(int size);
/*
    - 功能：在内核中创建一个 epoll 实例（由红黑树和就绪链表组成）
    - size：目前该参数只要大于 0 即可
        - 历史上：size 是告诉内核期望监控的 fd 数量，用于内核分配哈希表大小。
        - 现代（Linux 2.6.8 之后）：内核改用红黑树动态管理，size 只要 > 0，内核会自动忽略具体数值。
    - 返回值：
        - 成功：返回一个非负的文件描述符（epoll 文件描述符），用于后续操作。
        - 失败：-1，并设置 errno。
*/
```

```c
#include <sys/epoll.h>

int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
/*
    - 功能：对 epoll 实例进行控制（添加、修改、删除监控的文件描述符）
    - epfd：epoll_create 返回的 epoll 文件描述符
    - op：操作类型
        - EPOLL_CTL_ADD：注册新的 fd 到 epfd 中（添加进红黑树）
        - EPOLL_CTL_MOD：修改已经注册的 fd 的监听事件
        - EPOLL_CTL_DEL：从 epfd 中删除一个 fd（移出红黑树）
    - fd：需要操作的文件描述符（如 lfd 或 cfd）
    - event：-I，告诉内核需要监控什么事件（当op为EPOLL_CTL_DEL的时候，指定为NULL）
    - 返回值：
        - 成功：0
        - 失败：-1，并设置 errno
*/
```

其中

```c
#include <sys/epoll.h>

struct epoll_event {
    uint32_t     events;      /* Epoll 事件类型（位掩码） */
    epoll_data_t data;        /* 用户数据（通常存 fd 或自定义结构体指针） */
};

/* - events 常用取值：
        - EPOLLIN ：表示对应的文件描述符可以读（包括对端正常关闭）
        - EPOLLOUT：表示对应的文件描述符可以写
        - EPOLLPRI：表示有紧急的数据可读（如带外数据）
        - EPOLLERR：表示对应的文件描述符发生错误（自动监控，无需设置）
        - EPOLLHUP：表示对应的文件描述符被挂断（自动监控，无需设置）
        - EPOLLET ：将 epoll 设为边缘触发（Edge Triggered）模式
        - EPOLLONESHOT：一次性监听，触发后需再次设置才能继续监听
    - data：这是一个联合体（union），允许用户存储关联信息
*/
```

```c
#include <sys/epoll.h>

int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
/*
    - 功能：等待内核通知在 epoll 实例中登记过的 I/O 事件。
    - epfd：epoll_create 返回的 epoll 文件描述符。
    - events：-O，保存了发生了变化的文件描述符信息
    - maxevents：events 数组的大小
    - timeout：等待的超时时间（毫秒 ms）：
        - -1：永久阻塞，直到有 fd 就绪。
        - 0：立即返回（非阻塞轮询）。
        - >0：等待指定的毫秒数。
    - 返回值：
        - > 0：成功，返回就绪的文件描述符总数。
        - 0：超时，没有事件发生。
        - -1：失败，并设置 errno。
*/
```

### 2.3.3 代码实现

```c
#include <sys/epoll.h>
#include <sys/time.h>
#include <sys/types.h>
#include <unistd.h>
#include <sys/socket.h>
#include <stdio.h>
#include <arpa/inet.h>
#include <string.h>

#define SERVER_IP "172.16.0.6"
#define SERVER_PORT 9999

int main(void)
{
    /* 1. 创建socket */
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0)
    {
        perror("create listen fd error");
        return 1;
    }

    /* 2. 绑定端口 */
    int opt = 1;
    setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)); // 设置端口复用（放在bind前）

    struct sockaddr_in saddr;
    memset(&saddr, 0, sizeof(saddr));
    saddr.sin_family = AF_INET;
    inet_pton(AF_INET, SERVER_IP, &saddr.sin_addr);
    saddr.sin_port = htons(SERVER_PORT);
    if (bind(listen_fd, (struct sockaddr *)&saddr, sizeof(saddr)) == -1)
    {
        perror("socket bind error");
        close(listen_fd);
        return 1;
    }

    /* 3.开始监听 */
    if (listen(listen_fd, 10) == -1)
    {
        perror("socket listen error");
        close(listen_fd);
        return 1;
    }

    /* 4. 创建epoll */
    int epfd = epoll_create(1);
    if (epfd < 0)
    {
        perror("create epoll error.");
        close(listen_fd);
        return 1;
    }
    struct epoll_event epevent;
    epevent.events = EPOLLIN;
    epevent.data.fd = listen_fd;
    if (epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &epevent) < 0)
    {
        perror("add lfd to epoll error");
        return 1;
    }

    struct epoll_event epevents[512];        // 获取就绪队列的fd列表
    struct sockaddr_in client_addr;          // 客户端地址信息
    socklen_t addrlen = sizeof(client_addr); // 地址长度
    char recv_buf[1024];                     // cfd接收

    /* 5. 等待连接 */
    printf("服务器正在等待被连接...\n");
    while (1)
    {
        /* 6. 等待epoll中消息到达事件 */
        int ret = epoll_wait(epfd, epevents, 512, -1);
        if (ret < 0)
        {
            perror("epoll wait error.");
            continue;
        }
        else
        {
            printf("%d个fd待处理\n", ret);
            /* 7. 遍历数组 */
            for (int i = 0; i < ret; i++)
            {
                /* 8. 如果是lfd有数据，说明到达一个连接请求 */
                if (epevents[i].data.fd == listen_fd)
                {
                    /* 8-1. 连接客户端 */
                    socklen_t addrlen = sizeof(client_addr);
                    memset(&client_addr, 0, sizeof(client_addr));
                    int cfd = accept(listen_fd, (struct sockaddr *)&client_addr, &addrlen);
                    if (cfd < 0)
                    {
                        perror("accept function called error");
                        continue;
                    }
                    /* 8-2. 把cfd加入管理 */
                    epevent.data.fd = cfd;
                    epevent.events = EPOLLIN;
                    if (epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &epevent) < 0)
                    {
                        perror("cfd add to epoll error");
                        close(cfd);
                    }
                    char client_ip[16];
                    printf("连接成功，客户端ip：%s:%d\n", inet_ntop(AF_INET, &client_addr.sin_addr, client_ip, sizeof(client_ip)), ntohs(client_addr.sin_port));
                }
                /* 9. 如果cfd有数据，收发数据 */
                else
                {
                    int cfd = epevents[i].data.fd;
                    int len = recv(cfd, recv_buf, sizeof(recv_buf), 0);
                    if (len < 0)
                    {
                        perror("recv error");
                        close(cfd);
                        epoll_ctl(epfd, EPOLL_CTL_DEL, cfd, NULL);
                    }
                    else if (len == 0)
                    {
                        printf("fd%d连接关闭\n", cfd);
                        close(cfd);
                        epoll_ctl(epfd, EPOLL_CTL_DEL, cfd, NULL);
                    }
                    else
                    {
                        printf("fd%d接收：", cfd);
                        fflush(stdout);
                        write(1, recv_buf, len);
                        printf("\n");
                    }
                }
            }
        }
    }

    epoll_ctl(epfd, EPOLL_CTL_DEL, listen_fd, NULL);
    close(listen_fd);
    return 0;
}
```

### 2.3.4 两种工作模式

- **LT（Level Triggered，水平触发，默认模式）**：只要缓冲区里还有数据，就会一直通知你

  > **举个例子**
  >
  > 客户端发来 1000 字节。
  >
  > 你这次只读了 200 字节，还剩 800 字节在缓冲区里。
  >
  > 那么：
  >
  > - 第一次 `epoll_wait` 告诉你：这个 fd 可读
  > - 你读了 200 字节
  > - 下次再 `epoll_wait`
  > - 因为缓冲区里还剩 800 字节，所以**还会再次通知你**

  **宗旨**：只要“当前还能读”，就反复提醒你

  **优点**：容错高，不容易漏数据。

  **缺点**：同一个 fd，只要还有数据没处理完，`epoll_wait` 可能反复返回它。所以**效率**上不如 ET“干脆”。

- **ET（Edge Triggered，边缘触发）**：只在“状态变化的那一下”通知你一次

  > **还是刚才那个例子**
  >
  > 客户端发来 1000 字节。
  >
  > - 缓冲区原来空
  > - 现在变成有数据
  > - `epoll_wait` 返回一次
  >
  > 这次你只读了 200 字节，还剩 800 字节。
  >
  > 这时因为 fd 还是“可读状态”（缓冲区里还有数据），没有发生新的边沿变化，所以：
  >
  > - 下一次 `epoll_wait`
  > - **可能就不再返回这个 fd 了**
  >
  > 如果你不主动把剩下 800 字节继续读完，就可能一直留在那里。

---

**注意点**

- **ET模式下，必须配合`epoll_wait`的非阻塞方式**。

  因为用户不把缓冲区内的数据读完，缓冲区内一直有数据，一直是可读状态，之后每次数据来的时候产生不了边沿，内核无法通知，如果使用阻塞式，就会一直阻塞。

  所以在 ET 模式下，**一旦收到可读通知，就要一直读，读到不能再读为止。**

  - 循环 `read/recv`
  - 一直读到返回 `-1` 且 `errno == EAGAIN` 或 `EWOULDBLOCK`

---

**如何使用**

```c
struct epoll_event ev;
ev.events = EPOLLIN | EPOLLET;	// 设置该fd为边沿触发
ev.data.fd = fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);
```
