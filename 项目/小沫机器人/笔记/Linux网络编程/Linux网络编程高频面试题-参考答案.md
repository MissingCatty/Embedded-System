# Linux 网络编程高频面试题参考答案

说明：这是一份偏面试表达的参考答案，不追求教材式展开，重点是帮助你形成可口述、可追问的回答框架。

## 一、文件 IO 与基础能力

### 1. 标准 C 库 IO 和 Linux 系统 IO 有什么区别？为什么网络编程里更常用系统 IO？

答：标准 C 库 IO 是对系统 IO 的封装，比如 `fopen`、`fread`、`fwrite`，它带有用户态缓冲区，使用方便。Linux 系统 IO 是 `open`、`read`、`write` 这类系统调用，更接近内核。网络编程里更常用系统 IO，是因为 socket 本身就是文件描述符体系的一部分，配合 `read/write`、`send/recv`、`fcntl`、`epoll` 这些接口更自然，也更容易精确控制阻塞、非阻塞和事件驱动行为。

### 2. 什么是文件描述符？它和 `FILE*` 有什么区别？

答：文件描述符本质上是一个整数，是进程打开文件、管道、socket 等内核对象时得到的“索引”。`FILE*` 是 C 标准库层面的文件流指针，内部封装了文件描述符、缓冲区和读写状态。简单说，文件描述符更底层，`FILE*` 更高级、更方便，但网络编程通常直接操作文件描述符。

### 3. `open` 的常见标志位有哪些？`O_CREAT` 与权限位应该怎么理解？

答：常见标志位有 `O_RDONLY`、`O_WRONLY`、`O_RDWR`、`O_CREAT`、`O_TRUNC`、`O_APPEND`、`O_NONBLOCK`。`O_CREAT` 表示文件不存在就创建，这时第三个参数的权限位才有意义，比如 `0666`。最终权限还会受到 `umask` 的影响，所以不是你写多少就一定落成多少。

### 4. `read` / `write` 可能出现“短读/短写”吗？写代码时要注意什么？

答：可能。`read` 可能少于期望字节数返回，`write` 也可能只写出一部分，尤其是管道、socket、非阻塞 IO 更常见。所以不能假设一次 `read/write` 就完成全部数据传输，应该根据返回值循环处理，并对 `EINTR`、`EAGAIN` 这类情况做正确判断。

### 5. 如何把一个文件描述符设置成非阻塞？为什么通常用 `fcntl` 来做？

答：常见做法是先用 `fcntl(fd, F_GETFL)` 取出当前状态标志，再加上 `O_NONBLOCK`，最后用 `fcntl(fd, F_SETFL, flags)` 写回。之所以常用 `fcntl`，是因为它是 Linux/Unix 上通用的文件描述符控制接口，既能改阻塞属性，也能做复制描述符、获取状态等操作。

### 6. `dup` 和 `dup2` 的作用是什么？它们在重定向里为什么常用？

答：`dup` 是复制一个已有的文件描述符，返回当前可用的最小新 fd；`dup2` 是把一个 fd 复制到指定编号上。它们常用于重定向，是因为可以把标准输入输出错误重定向到文件、管道或 socket 上，比如把标准输出重定向到某个文件，后续 `printf` 的结果就会写入那个文件。

## 二、进程与进程控制

### 7. 进程和线程的核心区别是什么？

答：进程是资源分配的基本单位，线程是 CPU 调度的基本单位。一个进程有独立地址空间，不同进程之间默认彼此隔离；同一进程内的多个线程共享代码段、数据段、堆、打开的文件等资源，但各自有独立栈和寄存器上下文。简单说，进程更重、更隔离，线程更轻、更适合共享数据。

### 8. `fork` 的返回值规则是什么？为什么说它“调用一次，返回两次”？

答：`fork` 成功后会创建一个子进程。对父进程来说，返回值是子进程 PID；对子进程来说，返回值是 `0`；失败时只在父进程返回 `-1`。之所以说“调用一次，返回两次”，是因为 `fork` 之后父子进程都会从 `fork` 返回点继续执行。

### 9. `fork` 之后，父子进程哪些资源是共享的，哪些资源是独立的？

答：从逻辑上看，父子进程有各自独立的虚拟地址空间，所以代码段、数据段、堆、栈最终都是彼此独立的。它们会继承很多内核资源，比如打开的文件描述符、信号处理方式、环境变量等，但进程标识如 PID 一定不同。需要注意，很多资源在 `fork` 后一开始看起来一样，是因为底层用了写时拷贝。

### 10. 什么是写时拷贝（Copy On Write）？它解决了什么问题？

答：写时拷贝是 `fork` 后的一种优化策略。父子进程先共享同一份物理内存页，页被标成只读；谁先写，谁触发缺页异常，内核再为它复制新页。这样避免了 `fork` 时立刻复制整块地址空间，减少了时间和内存开销，尤其适合“`fork` 后马上 `exec`”这种常见场景。

### 11. `exec` 函数族的作用是什么？调用成功后，哪些东西会变，哪些东西通常不变？

答：`exec` 的作用是用一个新程序替换当前进程的用户态映像。调用成功后，原来的代码段、数据段、堆、栈都被新程序替换，程序从新程序的入口重新开始执行；但进程还是原来的进程，PID 通常不变。很多内核级属性会保留，比如部分打开的文件描述符也可能继续保留，除非设置了 `close-on-exec`。

### 12. `exit` 和 `_exit` 有什么区别？

答：`exit` 是 C 标准库函数，会先执行用户态清理逻辑，比如刷新 stdio 缓冲区、调用 `atexit` 注册函数，然后再退出。`_exit` 更底层，直接让进程退出，不做这些用户态收尾。面试里常见的点是：`fork` 后子进程如果只是失败退出，常更适合用 `_exit`，避免把父进程缓冲区重复刷一遍。

### 13. `wait` 和 `waitpid` 有什么区别？各自适合什么场景？

答：`wait` 只能阻塞等待任意一个子进程退出；`waitpid` 更灵活，可以指定等待哪个子进程，也可以用 `WNOHANG` 做非阻塞回收。单个子进程、简单同步时用 `wait` 就够了；多个子进程、需要指定目标或需要异步回收时更常用 `waitpid`。

### 14. 什么是孤儿进程？什么是僵尸进程？

答：孤儿进程是父进程先退出，而子进程还在运行，这时子进程会被 `init/systemd` 接管。僵尸进程是子进程已经退出，但父进程还没调用 `wait/waitpid` 回收它，导致它的 PCB 和退出状态还留在系统里。孤儿进程不是主要问题，僵尸进程才会持续占用 PID 等内核资源。

### 15. 为什么 `kill -9` 无法“杀死”僵尸进程？

答：因为僵尸进程本质上已经“死了”，只是退出信息还没被父进程回收。它不再真正运行，也不再调度，所以给它发信号没有意义。解决方法不是再杀它一次，而是让父进程执行 `wait/waitpid`，或者让父进程退出，由 `init/systemd` 接管并回收。

### 16. 如果父进程要回收大量子进程，通常会怎么设计？

答：常见做法是给父进程注册 `SIGCHLD` 处理逻辑，在处理逻辑里循环调用 `waitpid(-1, NULL, WNOHANG)`，一次把所有已退出子进程回收掉。这样可以避免父进程为某一个子进程阻塞，也能防止多个子进程几乎同时退出时漏回收。

### 17. 什么是守护进程？它和普通后台进程有什么区别？

答：守护进程是长期在后台运行、脱离控制终端的服务进程，比如日志服务、定时服务、Web 服务。普通后台进程只是 shell 里加个 `&` 放到后台，它仍可能依赖当前终端和会话；守护进程则需要主动脱离终端、会话和工作目录，更稳定，更适合作为系统服务。

### 18. 创建守护进程的典型步骤有哪些？每一步背后的目的是什么？

答：典型步骤是：先 `fork` 一次让父进程退出；子进程调用 `setsid` 创建新会话，脱离控制终端；再根据需要二次 `fork` 防止重新获得控制终端；修改工作目录到根目录，避免占用挂载点；重设 `umask`；关闭或重定向标准输入输出错误。核心目的就是脱离终端、避免受外部环境影响，并让它适合长期稳定运行。

## 三、进程间通信与信号

### 19. Linux 常见的进程间通信方式有哪些？分别适合什么场景？

答：常见 IPC 有管道、FIFO、消息队列、共享内存、信号、socket、本地 `mmap` 等。管道/FIFO 适合流式传输，写法简单；共享内存适合大数据量、高性能共享；信号适合事件通知，不适合传大量数据；socket 既能做本机通信，也能做跨主机通信，通用性最强。

### 20. 匿名管道的本质是什么？为什么它通常只能用于有血缘关系的进程？

答：匿名管道本质上是内核维护的一块缓冲区，同时给出读端和写端两个文件描述符。它通常只能用于有血缘关系的进程，是因为父子进程通过 `fork` 继承了同一组文件描述符，才能共同指向这根“管道”。无关进程没有这个共享的 fd 起点，所以一般用不了匿名管道。

### 21. 管道中，`read` 什么时候返回 `0`？`write` 什么时候会触发 `SIGPIPE`？

答：当管道写端全部关闭，并且管道缓冲区里的数据也已经读完时，`read` 返回 `0`，表示 EOF。若管道的所有读端都已经关闭，这时再对写端调用 `write`，进程通常会收到 `SIGPIPE`，同时 `write` 失败。面试里高频点是：父子进程必须及时关闭自己不用的那一端，否则 EOF 和 `SIGPIPE` 的行为都可能不符合预期。

### 22. 有名管道（FIFO）和匿名管道有什么区别？

答：匿名管道没有文件实体，通常只用于有血缘关系的进程；FIFO 在文件系统中有路径名，因此无关进程也可以通过同一路径打开它来通信。两者本质上都走管道语义，都是半双工、先进先出，只是 FIFO 多了一个文件系统入口。

### 23. `mmap` 能做什么？用它做进程间通信时有哪些前提和注意事项？

答：`mmap` 可以把文件或设备映射到进程地址空间里，实现“像访问内存一样访问文件”，也能用于共享数据。做 IPC 时常用 `MAP_SHARED`，并且文件大小不能是 0，必要时要先用 `ftruncate` 扩容。常见注意点有：`offset` 要按页对齐，`munmap` 必须传原始首地址，并且映射共享并不自动解决并发同步问题，仍需要锁或协议约束。

### 24. 共享内存的典型使用流程是什么？请结合 `shmget`、`shmat`、`shmdt`、`shmctl` 说明。

答：先通过 `ftok` 或约定好的 key 调用 `shmget` 获取或创建共享内存段，得到 `shmid`；再用 `shmat` 把这段共享内存映射到当前进程地址空间，拿到可访问指针；用完后调用 `shmdt` 解除映射；最后通过 `shmctl(shmid, IPC_RMID, NULL)` 标记删除。面试里要补一句：`IPC_RMID` 往往是“标记删除”，不是立刻物理消失，通常要等所有进程都分离后才真正释放。

### 25. 共享内存和管道相比，各自的优缺点是什么？

答：共享内存的优点是速度快、拷贝少，适合大数据量；缺点是自己要处理同步互斥，编程复杂度更高。管道的优点是简单、天然带顺序语义；缺点是数据要经过内核缓冲区搬运，吞吐和灵活性不如共享内存。面试里常见总结是：共享内存最快，但同步最难。

### 26. 什么是信号？信号通常由谁发出？默认处理动作有哪些？

答：信号可以理解成 Linux 里的软件中断，用来通知进程“某个事件发生了”。它可以由内核发出，比如子进程退出、除零、非法内存访问；也可以由其他进程或进程自己发出，比如 `kill`、`raise`。默认处理动作常见有终止、忽略、停止、继续、终止并产生 core 文件。

### 27. 哪些信号不能被捕获、阻塞或忽略？

答：最典型的是 `SIGKILL` 和 `SIGSTOP`。`SIGKILL` 用于强制终止，`SIGSTOP` 用于强制暂停，它们都是内核保留的强制控制信号，进程不能自定义处理逻辑。

### 28. `signal` 和 `sigaction` 有什么区别？为什么工程上更推荐 `sigaction`？

答：`signal` 接口更老、更简单，但不同系统上的语义历史上并不完全一致。`sigaction` 更标准、更可控，可以设置更详细的处理行为和屏蔽字，也更适合工程代码。面试里常见回答是：`sigaction` 功能更完整、行为更明确，所以生产环境优先用它。

### 29. 什么是信号屏蔽？什么是未决（挂起）信号？

答：信号屏蔽是指进程暂时不处理某些信号，可以通过 `sigprocmask` 修改屏蔽字。信号发生后，如果它当前被阻塞了，就不会立刻递达处理，而是进入未决集合，也就是挂起状态。等解除屏蔽后，内核才会把它真正递达给进程。

### 30. 为什么回收子进程时常见写法是 `SIGCHLD + waitpid(-1, NULL, WNOHANG)`？

答：因为 `SIGCHLD` 能在子进程退出时及时通知父进程，而 `waitpid(-1, NULL, WNOHANG)` 可以非阻塞地回收任意一个已退出子进程。再配合 `while` 循环，就能一次清掉多个同时退出的孩子，既不阻塞主流程，也不容易留下僵尸进程。

## 四、线程与同步

### 31. 多线程程序中，哪些资源是线程共享的，哪些资源是线程私有的？

答：同一进程内的线程共享代码段、数据段、堆、打开的文件描述符、全局变量等资源。每个线程私有的是栈、寄存器上下文、线程 ID、调度状态等。面试里容易追问的一点是：局部变量通常在线程自己的栈上，所以默认不是线程共享的。

### 32. `pthread_create`、`pthread_join`、`pthread_detach` 分别解决什么问题？

答：`pthread_create` 用来创建线程；`pthread_join` 用来等待线程结束并回收线程资源，还能拿到线程返回值；`pthread_detach` 是把线程设成分离态，线程结束后资源自动回收，不再允许 `join`。简单说，`join` 适合需要等待结果，`detach` 适合“干完就自动走”的后台工作线程。

### 33. 线程有哪些常见退出方式？

答：常见方式有：线程函数 `return`；调用 `pthread_exit`；被其他线程通过 `pthread_cancel` 取消；所属进程整体退出时线程一起结束。面试时可以补一句，`main` 线程如果直接 `return` 导致整个进程结束，其他线程也会一起被终止。

### 34. 为什么多线程程序会产生竞态条件？你怎么向面试官解释“卖票问题”？

答：因为多个线程会并发访问同一份共享数据，而“判断 + 修改 + 输出”这类操作往往不是原子的。卖票问题里，多个线程都看到票数大于 0，然后交替执行，可能出现卖出 0 号票、重复卖同一张票等错误。核心原因就是共享变量访问缺少同步保护。

### 35. 互斥锁的作用是什么？使用互斥锁时最常见的错误有哪些？

答：互斥锁用于保护临界区，保证同一时刻只有一个线程能访问共享资源。常见错误有：忘记解锁；同一线程重复加普通锁导致自锁；临界区过大导致性能差；异常路径提前返回却没解锁。面试里要体现出一句：互斥锁保护的是“共享资源的一致性”，不是为了让程序“有序打印”。

### 36. 死锁常见的产生原因有哪些？如何避免？

答：常见原因有忘记解锁、同一线程重复锁同一把非递归锁、多线程拿多把锁时顺序不一致。避免方式通常是统一加锁顺序、缩小锁粒度、使用超时/尝试加锁、在异常路径确保释放资源。面试里如果能提到“破坏死锁四必要条件”会更完整。

### 37. 读写锁适合什么场景？它和普通互斥锁相比有什么特点？

答：读写锁适合“读多写少”的场景。它允许多个读线程并发持有读锁，但写锁是独占的，写期间不允许其他线程读写。同样是保护共享资源，读写锁在读多写少时吞吐通常比普通互斥锁更好，但实现和调度成本也更高。

### 38. 信号量和互斥锁有什么区别？

答：互斥锁更强调“互斥访问”和“所有权”，通常是谁加锁谁解锁，值本质上只有锁住和解锁两种状态。信号量本质是计数器，表示可用资源数量，可以用来做资源配额控制，也常用在生产者消费者模型里做同步。简单说，互斥锁偏保护临界区，信号量偏表示资源个数和先后关系。

### 39. 条件变量解决的是什么问题？为什么它通常要配合互斥锁，并且常配合 `while` 使用？

答：条件变量用于“等待某个条件成立”，比如队列非空、队列未满。它本身不保护共享数据，所以要和互斥锁一起用，先锁住共享状态，再检查条件。之所以常用 `while` 而不是 `if`，是为了防止伪唤醒，或者避免被唤醒后条件又被其他线程先一步破坏。

### 40. 如果让你实现“生产者-消费者模型”，你会如何设计同步方案？

答：常见设计是一个共享队列，外加一把互斥锁保护队列；再用两个条件变量或两个信号量表示“队列非空”和“队列未满”。生产者在队列满时等待，消费者在队列空时等待。面试里如果结合线程池，可以进一步说主线程是生产者，工作线程是消费者。

## 五、网络基础与 TCP/UDP

### 41. MAC 地址和 IP 地址分别解决什么问题？交换机和路由器分别工作在什么层？

答：MAC 地址主要用于同一链路、同一局域网内的帧转发，IP 地址用于跨网络寻址和路由。交换机主要看二层 MAC 头，工作在数据链路层；路由器需要看三层 IP 头，工作在网络层。简单说，MAC 解决“局域网里交给谁”，IP 解决“跨网要送到哪台主机”。

### 42. TCP/IP 分层模型里，各层分别负责什么？

答：应用层定义应用数据格式和协议语义，比如 HTTP、DNS；传输层负责进程到进程的通信，比如 TCP/UDP、端口、可靠性；网络层负责 IP 寻址和路由；链路层负责把数据在当前链路上发给下一跳。面试里不一定拘泥于四层还是五层，关键是能说清职责边界。

### 43. `0.0.0.0` 和 `127.0.0.1` 有什么区别？在服务器编程中分别怎么用？

答：`127.0.0.1` 是回环地址，表示“本机自己访问自己”，不会经过真实网卡。`0.0.0.0` 在服务器监听场景里通常表示绑定本机所有网卡地址。比如服务端 `bind` 到 `0.0.0.0:8080`，表示外部只要能到达本机任一网卡，都可以连上这个服务。

### 44. 为什么网络编程里要做字节序转换？`htonl`、`htons`、`ntohl`、`ntohs` 分别用在什么地方？

答：因为不同主机的本地字节序可能不同，而网络字节序通常约定为大端，所以多字节整数在发送前要转成统一格式。`htons/ntohs` 用于 16 位数据，常见是端口；`htonl/ntohl` 用于 32 位数据，常见是 IPv4 地址或其他 32 位整数。统一字节序的目的是保证不同机器解释同一串字节时结果一致。

### 45. TCP 和 UDP 的核心区别是什么？各自适合哪些场景？

答：TCP 是面向连接、可靠、按序、字节流协议，有重传、流量控制和拥塞控制；UDP 是无连接、尽力而为、面向报文，开销更小。TCP 适合文件传输、Web、数据库这类对可靠性要求高的场景；UDP 适合音视频、实时游戏、广播发现等更看重时延和简单性的场景。

### 46. 请说一下 TCP 三次握手的过程。为什么不能只握两次？

答：第一次，客户端发 `SYN`，表示想建立连接；第二次，服务端回 `SYN+ACK`，表示我收到了，也同意建立；第三次，客户端再回 `ACK`，表示我也收到了你的确认。不能只握两次，是因为服务端需要确认“自己的发送能力”和“客户端的接收能力”也没问题，同时第三次握手还能避免某些历史连接请求报文导致服务端误建连接。

### 47. 请说一下 TCP 四次挥手的过程。为什么通常是四次而不是三次？

答：主动关闭方先发 `FIN`，表示自己没数据发了；对方回 `ACK`，表示知道了；等对方把剩余数据发完后，再发自己的 `FIN`；最后主动关闭方回 `ACK`。之所以通常是四次，是因为 TCP 是全双工的，双方的“发送方向”要分别关闭，所以确认对方不再发送和自己不再发送通常不是同一个时刻。某些情况下中间两步可能合并，但原理上是两次方向独立关闭。

### 48. 什么是 `TIME_WAIT`？它为什么存在？为什么主动关闭连接的一方更容易进入它？

答：`TIME_WAIT` 是连接关闭后的一种等待状态，持续时间通常是 2MSL。它的意义主要有两个：一是保证最后一个 `ACK` 丢了时还能重传；二是让旧连接中的延迟报文在网络里自然消失，避免污染新连接。通常主动关闭的一方发送最后一个 `ACK`，所以更容易进入 `TIME_WAIT`。

### 49. 如果线上服务器出现大量 `CLOSE_WAIT`，你首先会怀疑什么问题？

答：首先会怀疑应用层代码没有及时关闭 socket。`CLOSE_WAIT` 说明对端已经发了 `FIN`，本端也收到了，但本端还没调用 `close/shutdown` 把连接彻底关掉。大量 `CLOSE_WAIT` 往往不是内核问题，而是业务线程卡住、异常路径漏关连接或者资源释放逻辑有 bug。

### 50. 什么是半关闭？`shutdown` 和 `close` 的区别是什么？

答：半关闭是指只关闭连接的一半方向，比如只关闭写端但保留读端。`shutdown(sockfd, SHUT_WR)` 后，本端不再发送数据，会发 `FIN`，但还可以继续接收对端数据；`close` 关闭的是文件描述符引用，只有当该连接最后一个引用被关掉时，底层连接才真正关闭。工程上上传文件后等待服务端回执，就是半关闭的典型场景。

### 51. TCP 为什么会出现粘包/拆包？常见解决方案有哪些？

答：因为 TCP 是字节流，没有消息边界，发送端多次写入的数据可能被合并，接收端一次读也可能只读到半包。解决方案通常是在应用层自行定义边界，比如固定长度协议、分隔符协议、消息头加长度字段的 length-field 协议。面试里最好补一句：粘包不是 TCP 的 bug，而是应用层消息边界要自己设计。

### 52. TCP 的流量控制和拥塞控制有什么区别？

答：流量控制是端到端的，目的是别把接收方压垮，核心依据是接收窗口 `rwnd`；拥塞控制是面向整个网络的，目的是别把网络链路和路由器压垮，核心依据是拥塞窗口 `cwnd`。前者关注“对方来不来得及收”，后者关注“网络现在堵不堵”。

## 六、Socket 编程与 IO 多路复用

### 53. TCP 服务器和客户端的基本 socket 通信流程分别是什么？

答：服务器通常是 `socket -> bind -> listen -> accept -> recv/send -> close`。客户端通常是 `socket -> connect -> recv/send -> close`。面试里要强调一点：服务器有一个监听 socket，还有多个已连接 socket，真正业务收发通常发生在已连接 socket 上。

### 54. `bind`、`listen`、`accept`、`connect` 各自扮演什么角色？

答：`bind` 是给 socket 绑定本地 IP 和端口；`listen` 是把 TCP socket 变成监听状态；`accept` 是从已完成连接队列里取出一个新连接，返回新的已连接 fd；`connect` 是客户端主动向服务端发起连接。可以把它们理解成：先“占门牌号”，再“开门营业”，再“接待客人”。

### 55. 为什么 `accept` 会返回一个新的已连接套接字，而监听套接字不能直接用来收发业务数据？

答：因为监听套接字的职责是持续接收新的连接请求，它对应的是“服务入口”；每个客户端建立连接后，都需要一个独立的连接状态和收发缓冲区，所以内核会为每个连接创建新的已连接 socket。这样监听 fd 继续负责接待新连接，业务 fd 负责和具体客户端通信，职责才清晰。

### 56. `setsockopt(... SO_REUSEADDR ...)` 解决的是什么问题？为什么必须在 `bind` 前设置？

答：它主要解决的是端口因为旧连接处于 `TIME_WAIT` 导致服务重启时 `bind` 失败的问题。必须在 `bind` 前设置，是因为内核在绑定端口时就要决定是否允许这次绑定，如果你绑定完再改选项，时机已经错过了。

### 57. 阻塞 IO、非阻塞 IO、IO 多路复用三者分别是什么思路？

答：阻塞 IO 是没数据就睡在那；非阻塞 IO 是没数据立刻返回，应用自己反复轮询；IO 多路复用是把“等谁就绪”这件事交给内核统一管理，应用线程只在有 fd 就绪时再去处理。高并发场景下，IO 多路复用的价值在于减少线程数量和无效轮询。

### 58. Linux 中常说的五种 IO 模型分别是什么？

答：通常指阻塞 IO、非阻塞 IO、IO 多路复用、信号驱动 IO、异步 IO。前四种本质上在“数据从内核拷到用户缓冲区”这个阶段通常还是同步的，只有异步 IO 才是连数据搬运都由内核完成后再通知应用。

### 59. `select`、`poll`、`epoll` 的核心区别是什么？

答：`select` 用位图管理 fd，数量通常受 `FD_SETSIZE` 限制，而且每次调用都要把整套集合从用户态拷到内核态；`poll` 把位图换成了数组，解决了 1024 限制，但仍然要线性遍历、反复拷贝；`epoll` 在内核里维护监听集合和就绪队列，应用只关心活跃事件，更适合大量连接场景。

### 60. `select` 的主要缺点有哪些？

答：主要缺点有三个：一是 fd 数量通常受 1024 限制；二是每次调用前后都要在用户态和内核态之间拷贝 fd 集合；三是返回后应用还得自己 O(n) 遍历全部 fd 才知道到底谁就绪了。所以连接数一大，`select` 的性能和可维护性都比较差。

### 61. 为什么说 `epoll` 更适合高并发场景？

答：因为 `epoll` 把监听集合放在内核里维护，不需要每次都把整套 fd 列表从用户态拷来拷去；它返回的是就绪事件列表，而不是让你再去扫全量 fd；同时它没有 `select` 那种 1024 的固定上限。总结起来就是：活跃连接少但总连接多的时候，`epoll` 优势很明显。

### 62. `epoll` 的 LT 和 ET 两种工作模式有什么区别？ET 模式下为什么通常要配合非阻塞 IO？

答：LT 是水平触发，只要 fd 处于可读/可写状态，没处理完就会持续通知；ET 是边缘触发，只在状态从“无”变“有”的那一瞬间通知一次。ET 下如果你没一次把数据读干净，后面可能就收不到再次提醒，所以通常要把 socket 设成非阻塞，并在事件到来后循环读写直到返回 `EAGAIN`。

## 七、Web 服务器与高并发设计

### 63. HTTP 请求报文的基本结构是什么？

答：HTTP 请求报文通常由请求行、请求头、空行、请求体组成。请求行里有方法、URL、协议版本；请求头是一组键值对；空行表示头结束；请求体承载表单、JSON、文件等内容。不是所有请求都有请求体，比如 GET 通常没有。

### 64. GET 和 POST 有哪些常见区别？

答：最常见区别是语义不同：GET 更偏获取资源，POST 更偏提交数据。GET 参数常放在 URL 查询串里，POST 数据通常放在请求体里。工程上还常说 GET 更适合幂等查询，POST 更适合新增或提交，但真正语义还是看接口设计，不是“GET 一定安全、POST 一定不安全”这种绝对说法。

### 65. 什么是 Reactor？什么是 Proactor？两者的核心区别是什么？

答：Reactor 的思路是“事件来了，应用自己去做 IO”，比如 `epoll_wait` 告诉你哪个 fd 可读，然后线程再去 `read`；Proactor 的思路是“IO 已经由内核做完了，应用只处理结果”。核心区别就在于事件通知发生时，IO 是否已经完成。Linux 常见高并发网络程序更多是基于同步 IO 的 Reactor 风格。

### 66. 为什么高并发服务器常见架构是 `epoll + 非阻塞 socket + 线程池`？

答：`epoll` 负责高效发现就绪连接，非阻塞 socket 避免单个连接把线程卡死，线程池负责把耗时业务逻辑并行化处理。这样主线程专注于事件监听与分发，工作线程专注于具体任务执行，既能扛住大量连接，又能避免“一连接一线程”带来的上下文切换和资源消耗。

### 67. 线程池通常由哪些核心部分组成？主线程和工作线程一般如何同步？

答：线程池核心通常包括线程集合、任务队列、互斥锁、条件变量或信号量、停止标志等。主线程把任务放进队列，相当于生产者；工作线程从队列中取任务执行，相当于消费者。同步的关键点是：队列访问要互斥，队列空和满的状态要通过条件变量或信号量协调。

### 68. 如果让你口头设计一个简化版高并发 TCP/HTTP 服务器，你会怎么介绍整体架构？

答：我会说采用 Reactor 思路：一个主线程负责创建监听 socket，设置 `SO_REUSEADDR` 和非阻塞，使用 `epoll` 统一监听监听 fd 和所有连接 fd。新连接到来时 `accept` 并注册到 `epoll`；读事件触发后把连接或请求对象投递到线程池，工作线程做 HTTP 解析、业务处理和响应生成；写事件就绪时再把响应回写到 socket。为了稳定性，还要补上连接超时回收、异常关闭处理、半包粘包处理、日志和资源释放。面试里这样回答，通常已经能覆盖“网络模型 + 并发模型 + 工程细节”三层内容。
