# 1. BASEPRI属于一个任务的上下文

在做第4章验证中断关闭的实验时，文档中使用的延时函数为`delay_xms`，该函数**不会引起任务调度**。但我在实现时，偷了个懒，由于我自己没有实现`delay_xms`，就直接使用`vTaskDelay()`函数进行延时。

**非预期现象**：只要调用了`vTaskDelay`函数，就会造成关中断函数`portDISABLE_INTERRUPTS`失效。

**原因**：`portDISABLE_INTERRUPTS()` **只对当前运行的任务有效**。

```c
void interrupt_task(void *p_arg)
{
    while (1)
    {
        // 中断关闭五秒
        portDISABLE_INTERRUPTS();
        vTaskDelay(5000);
        // 中断开启五秒
        portENABLE_INTERRUPTS();
        vTaskDelay(5000);
    }
}
```

`portDISABLE_INTERRUPTS`函数实际上是设置寄存器`basepri`的值，将`basepri`设置为RTOS能够维护的最大优先级（源码中指定的最大优先级为5），等于说优先级数值大于`basepri`的中断（数值越大优先级越低）都被屏蔽。

而恰好`basepri`是一个任务的上下文，换言之，当进行任务调度时，当前任务`interrupt_task`的`basepri`值会被保存，切换到`other_task`（其他任务或IDLE任务）时，`basepri`寄存器就变为另一个任务的`basepri`值（如果任务中没有设置，默认为0）。

可以编写一个程序验证，在`interrupt_task`中，首先打印关中断后的`basepri`值，再在`other_task`中打印任务切换后的`basepri`值：

```c
// interrupt任务
void interrupt_task(void *p_arg)
{
    while (1)
    {
        // 中断关闭五秒
        portDISABLE_INTERRUPTS();
        usart_send_uint32(__get_BASEPRI() >> 4);
        tag = 0;
        usart_send_str("Interrupt Close.\r\n");
        vTaskDelay(5000);
    }
}

// 其他任务
void other_task(void *p_arg)
{
    while (1)
    {
        if (!tag)
        {
            usart_send_uint32(__get_BASEPRI() >> 4);
            tag = 1;
        }
    }
}
```

结果如下，可以看到当关中断后，`basepri`的值为5，当调用`vTaskDelay`后，`interrupt`任务被阻塞，触发任务切换，切换为`other_task`，在`other_task`中打印`basepri`的值为0。两次结果不一样。

![image-20250404192922247](https://zyc-learning-1309954661.cos.ap-nanjing.myqcloud.com/machine-learning-pic/19-29-29-e18cb6dae6f22236fc58f785d1e2ffba-image-20250404192922247-e9e389.png)

所以，延时函数不能触发任务调度，使用普通的定时器延时函数即可。

# 2 中断函数中不能使用任务挂起函数

在任务挂起恢复实验中，如果在按键的中断函数中使用譬如`vTaskSuspend`、`vTaskResume`、`eTaskGetState`等**所有用到`taskEnter_Critical`（进入临界区）都是不允许的**，即这些函数不能在中断里使用。

所以，用按键中断实现任务的挂起是不现实的，只能在另一个任务（如`key_task`）里使用轮询的方式去检查按键是否按下，再执行任务挂起操作。不过恢复函数有一个能够在中断中使用的版本`xTaskResumeFromISR`，使用该函数能够在中断函数中恢复某个任务。

# 3 中断优先级不能超过RTOS可管理的最大优先级

当在一个中断函数中对RTOS执行某些操作时，例如调用`vTaskResumFromISR()`，该中断的优先级不能比RTOS可管理的最大优先级还要大（比如RTOS管理的优先级数值大于等于5，那么中断的优先级就不能配置为0~4）

# 4 start_task任务的优先级务必最高

看一段程序

```c
#define START_TASK_PRIO 1                    // 任务优先级
#define START_STK_SIZE  128                  // 任务堆栈大小
TaskHandle_t StartTask_Handler;              // 任务句柄
void         start_task(void *pvParameters); // 任务函数

// 低优先级任务
#define LOW_TASK_PRIO 2                  // 任务优先级
#define LOW_STK_SIZE  256                // 任务堆栈大小
TaskHandle_t LowPrioTask_Handler;        // 任务句柄
void         low_prio_task(void *p_arg); // 任务函数

// 高优先级任务
#define HIGH_TASK_PRIO 3                  // 任务优先级
#define HIGH_STK_SIZE  256                // 任务堆栈大小
TaskHandle_t HighPrioTask_Handler;        // 任务句柄
void         high_prio_task(void *p_arg); // 任务函数

int main(void)
{
    usart_init(115200);
    led_init(&led0);
    led_init(&led1);

    // 创建互斥量
    xUARTSemaphore = xSemaphoreCreateMutex();
    
    xTaskCreate(
        (TaskFunction_t)start_task,        // 任务函数
        (const char *)"start_task",        // 任务名称
        (uint16_t)START_STK_SIZE,          // 任务堆栈大小
        (void *)NULL,                      // 传递给任务函数的参数
        (UBaseType_t)START_TASK_PRIO,      // 任务优先级
        (TaskHandle_t *)&StartTask_Handler // 任务句柄
    );
    vTaskStartScheduler(); // 启动任务调度
}

void start_task(void *pvParameters)
{
    // 注册子任务
    xTaskCreate(
        (TaskFunction_t)low_prio_task,
        (const char *)"low_prio_task",
        (uint16_t)LOW_STK_SIZE,
        (void *)NULL,
        (UBaseType_t)LOW_TASK_PRIO,
        (TaskHandle_t *)&LowPrioTask_Handler
    );

    xTaskCreate(
        (TaskFunction_t)high_prio_task,
        (const char *)"high_prio_task",
        (uint16_t)HIGH_STK_SIZE,
        (void *)NULL,
        (UBaseType_t)HIGH_TASK_PRIO,
        (TaskHandle_t *)&HighPrioTask_Handler
    );

    vTaskDelete(StartTask_Handler); // 删除开始任务
}
```

上面这段程序是错误的，因为在执行`start_task`时，第一个子任务创建完成后，在 `xTaskCreate` 函数**执行完成返回前的最后一刻**，如果它创建的新任务的优先级**高于**当前正在运行的任务，FreeRTOS 调度器**会立即强制执行一次任务切换（上下文切换）**。此时，任务执行权就交给了子任务`low_prio_task`，然后`start_task`就卡住了，一直没法创建高优先级的任务`high_prio_task`，同时`start_task`也没法被删除。
