# 1.内核、驱动和设备树的关系

一句话总结：内核是整体框架，驱动是功能实现，设备树是硬件描述。

## 1.1 内核

Linux 内核是操作系统的核心，负责：

- 进程管理
- 内存管理
- 文件系统
- 网络
- 设备驱动管理
- 中断处理

在设备管理方面，内核提供：

- 设备模型（device model）
- 总线框架（platform、i2c、spi、pci 等）
- 驱动注册与匹配机制

但内核本身**并不知道具体板子上有什么硬件**。

这就需要设备树。

## 1.2 设备树

设备树是：一种用来“描述硬件”的数据结构。

它告诉内核：

- 板子上有什么外设
- 这些外设挂在哪个总线上
- 用了哪些 GPIO
- 中断号是多少
- 寄存器地址是多少
- 时钟是多少

设备树本质是：

```
.dts → 编译 → .dtb（二进制）→ 内核启动时加载
```

举个例子：

```
i2c@40005400 {
    status = "okay";

    temperature@48 {
        compatible = "ti,tmp102";
        reg = <0x48>;
    };
};
```

意思是：

- 有一个 I2C 控制器
- I2C 总线上有个 TMP102 温度传感器
- 地址是 0x48

## 1.3 驱动

驱动是：控制某种硬件设备的代码。

比如：

- LED 驱动
- I2C 控制器驱动
- SPI Flash 驱动
- LCD 驱动
- 网卡驱动

驱动里面会写：

```
static const struct of_device_id tmp102_of_match[] = {
    { .compatible = "ti,tmp102" },
    { }
};
```

这里的 `"ti,tmp102"` 就是用来和设备树匹配的。

## 1.4 三者如何协同

### 内核启动

- Bootloader 把内核 + dtb 加载到内存
- 内核启动时解析 dtb

### 内核创建设备

内核会根据设备树：

- 创建 platform_device
- 创建设备结构体

### 驱动注册

驱动在内核中注册：

```c
platform_driver_register(&xxx_driver);
```

### 匹配机制

内核会根据：

```
设备树的 compatible
        ↓
驱动的 of_match_table
```

进行匹配。

如果匹配成功：

```
调用驱动的 probe() 函数
```

### probe 的作用

当匹配成功时：

```
static int xxx_probe(struct platform_device *pdev)
{
    // 1. 获取寄存器地址
    // 2. 申请中断
    // 3. 申请 GPIO
    // 4. 初始化硬件
}
```

此时：

✔ 驱动真正开始控制硬件
 ✔ 设备开始工作

# 2.编译内核

为什么编译驱动程序之前要先编译内核？ 

简单来说，驱动程序是基于内核代码编写的，需要调用内核中的：

- 内核内部函数
- 内核数据结构
- 内核宏定义
- 内核子系统接口

这些东西在不同内核版本中会变化。

所以驱动需要使用到某个版本的Linux内核源码，**编译驱动程序的Linux内核版本要和部署驱动程序的内核版本要一致**。

## 2.1 编译内核配置文件

不 同 的 开 发 板 对 应 不 同 的 配 置 文 件 ， 配 置 文 件 位 于 内 核 源 码arch/arm/configs/目录。 kernel 的编译过程如下：  

```bash
book@100ask:~$ cd 100ask_stm32mp157_pro-sdk/Linux-5.4/ 
book@100ask:~/100ask_stm32mp157_pro-sdk/Linux-5.4$ make 100ask_stm32mp157_pro_defconfig
```

> 在 Linux 内核源码里，有很多默认配置文件，放在：
>
> 在 Linux 内核源码里，有很多默认配置文件，放在：
>
> ```
> arch/arm/configs/
> ```
>
> 比如你这个板子对应的文件大概率是：
>
> ```
> arch/arm/configs/100ask_stm32mp157_pro_defconfig
> ```
>
> 当你执行：
>
> ```
> make 100ask_stm32mp157_pro_defconfig
> ```
>
> 本质就是：把 `arch/arm/configs/100ask_stm32mp157_pro_defconfig` 复制并展开生成当前目录下的 `.config`，也就是生成：`Linux-5.4/.config`
>
> `.config` 是：**内核的最终编译配置文件**。
>
> 它决定：
>
> - 编译哪些驱动
> - 是否启用网络
> - 是否支持 ext4
> - 是否支持 USB
> - 是否支持 I2C
> - 是否支持模块
>
> 里面的内容是：
>
> ```
> CONFIG_I2C=y
> CONFIG_USB=m
> CONFIG_EXT4_FS=y
> CONFIG_MODULES=y
> ```
>
> - y = 编进内核
> - m = 编译成模块
> - n = 不编译
>
> **为什么不能直接 make？**
>
> 如果你直接：
>
> ```
> make
> ```
>
> 会报错：
>
> ```
> No rule to make target
> ```
>
> 因为：内核在编译前必须有 .config，而 `.config` 就是通过 `xxx_defconfig` 生成的。
>
> defconfig 可以理解成：板子厂商帮你选好的默认菜单配置。它相当于提前帮你在 menuconfig 里点好了一堆选项。
>
> 你可以理解成：defconfig = 快速初始化配置
>
> 而当你执行：
>
> ```
> make menuconfig
> ```
>
> 会出现一个蓝色界面（基于 ncurses）
>
> 你可以：
>
> - 开关驱动
> - 打开文件系统支持
> - 选择是否编译成模块
> - 选择 CPU 类型
> - 开启调试选项
>
> 修改完退出时：
>
> 它会更新 `.config`
